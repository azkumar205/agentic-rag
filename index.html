BEGINNER Core Concepts

What is RAG?

Retrieval-Augmented Generation (RAG) is a pattern that combines a retrieval system with a generative AI model. Instead of relying solely on the LLM's training data, RAG first retrieves relevant documents from your knowledge base, then passes them to the LLM as grounding context to generate an accurate, cited answer.

User Query → Embed Query → Vector Search → Retrieve Chunks → Prompt LLM → Grounded Answer

What are Embeddings?

Embeddings are numerical vectors (arrays of floats) that represent the semantic meaning of text. Two semantically similar texts will have vectors that are close together in high-dimensional space, even if they use different words. This enables semantic search.

Example in this project:

// Generate embedding for a user query
float[] queryEmbedding = await _embeddingService.GenerateEmbeddingAsync(
    "What are the Q3 revenue figures?");
// Embedding model: text-embedding-ada-002 (1536 dimensions)

What is a Citation?

A citation tracks where a piece of retrieved information came from. It includes the file name, page number, section, and a text snippet. Users can download citation bundles as JSON for audit and traceability.

INTERMEDIATE Chunking, Hybrid Search & Incremental Ingestion

Document Chunking Strategy

Large documents cannot fit into LLM context windows or embedding models. We split them into smaller chunks with a sliding window overlap to preserve context at chunk boundaries.

Chunking parameters (DocumentExtractionService.cs):

  • Chunk size: 500 words
  • Overlap: 50 words (ensures boundary context is not lost)
  • Tables are extracted as standalone chunks (preserve structure)

Hybrid Search

Hybrid search combines keyword (BM25) and vector semantic search, then merges results using Reciprocal Rank Fusion (RRF). This combines the precision of keyword matching with the semantic understanding of vector search.

Search TypeMechanismBest For
SemanticPure vector similarity (cosine)Conceptual / meaning-based queries
KeywordBM25 full-text searchExact terms, product names, codes
HybridVector + BM25 fused with RRFMost queries — best overall recall

Weekly Incremental Ingestion

The system does incremental ingestion — only processing new or updated documents since the last run. This reduces cost and processing time vs. full reindexing.

Get last run timestamp → List new blobs (since timestamp) → Download & Extract → Chunk & Embed → Index to Azure AI Search → Update timestamp

The API endpoint POST /api/ingestion/run accepts an optional since timestamp. In production, an Azure Function timer trigger would call this weekly.

ADVANCED Agentic RAG, Tool Orchestration & Memory

What is Agentic RAG?

In classic RAG, the system follows a fixed pipeline: embed → retrieve → generate. In agentic RAG, the LLM acts as a reasoning agent that decides which tools to invoke, in what order, and how to combine results — all dynamically, based on the query intent.

AspectClassic RAGAgentic RAG
PipelineFixed (embed → retrieve → generate)Dynamic (LLM reasons and selects tools)
Data SourcesSingle (vector index)Multiple (docs, SQL, web)
Reasoning Steps1Up to N (configurable)
Tool UseNoYes (function calling)
MemoryNoYes (preferences, facts)
CachingOptionalBuilt-in (query + context)

Agent Tools

🔍 DocumentSearchTool

Performs hybrid search over indexed documents in Azure AI Search. Returns top-K chunks with file, page, section, and content.

🗄️ SqlQueryTool

Executes read-only SELECT queries against Azure SQL Server. Only SELECT statements permitted (security guardrail). Returns row data as JSON.

🌐 WebSearchTool

Queries the Bing Web Search API for current public information. Used when documents or SQL don't cover the query.

Agent Reasoning Loop (ReAct Pattern)

for step in 1..MaxReasoningSteps:
    response = LLM.complete(messages, tools=all_tools)

    if response.finish_reason == STOP:
        return final_answer + citations

    if response.finish_reason == TOOL_CALLS:
        for each tool_call in response.tool_calls:
            output = tool.execute(tool_call.args)
            messages.append(tool_result(output))
            extract_citations(output)

return "Max steps reached"

Caching Strategy

The CacheService uses IMemoryCache to avoid re-running expensive LLM calls for identical or near-identical queries.

  • Cache key: normalized query string (rag:{query.toLower().trim()})
  • Default TTL: 60 minutes (configurable via Agent:CacheExpirationMinutes)
  • Cached AgentResponse.FromCache = true to indicate source

Memorization

The MemoryService persists user preferences and important facts across sessions. This allows the system prompt to be personalized on subsequent requests.

// Store a preference
POST /api/memory/preferences
{ "userId": "user1", "key": "preferredLanguage", "value": "English" }

// Store an important fact
POST /api/memory/facts
{ "userId": "user1", "fact": "User works in the finance department" }

// These are injected into the system prompt on every agent call

Chat Thread Design

Each chat session has a unique thread ID. The agent maintains message history within the thread, enabling context carry-over across turns. Threads can be reset without losing the thread ID.

// Create thread
POST /api/chat/threads  →  { "id": "3f2a...", "title": "Thread 2025-..." }

// Send message (uses thread history as context)
POST /api/chat/threads/{threadId}/messages
{ "content": "What was mentioned in the Q3 report?" }

// Reset thread history (keep thread ID)
POST /api/chat/threads/{threadId}/reset

Security & Non-Functional Concerns

🔒 Security

  • SQL tool: SELECT-only (injection guardrail)
  • All secrets via environment variables / Azure Key Vault
  • HTTPS enforced on App Service
  • TLS 1.2 minimum on SQL and Storage
  • No public blob access

📈 Scalability

  • Stateless API — horizontal scale via App Service
  • Batch embedding generation reduces API calls
  • Azure AI Search partitions for indexing scale
  • In-memory cache reduces LLM calls

🛡️ Guardrails

  • Hallucination: LLM instructed to cite sources, avoid speculation
  • Prompt injection: System prompt not user-controllable
  • Data leakage: SQL read-only, no PII in citations
  • Reasoning step limit: prevents infinite loops

Interview-Ready Content

BEGINNER Questions

Q: What is the difference between classic RAG and agentic RAG?

A: Classic RAG follows a fixed pipeline (embed → retrieve → generate) with a single retrieval step and one data source. Agentic RAG uses an LLM as a reasoning agent that can dynamically select and chain multiple tools (document search, SQL, web search) over multiple reasoning steps to answer complex, multi-source queries.

Q: What is an embedding and why is it needed for RAG?

A: An embedding is a dense numerical vector that represents the semantic meaning of text. RAG needs embeddings to convert both documents and queries into the same vector space so that semantically similar content can be found via vector similarity (cosine distance), enabling queries to find relevant documents even when exact keyword matches don't exist.

Q: What is multimodal RAG?

A: Multimodal RAG processes different types of content beyond plain text — including images, tables, and scanned documents (via OCR). In this project, Azure Document Intelligence extracts text, tables, and OCR content from PDFs, Word docs, and images before chunking and indexing.

INTERMEDIATE Questions

Q: What is hybrid search and why is it better than pure vector search?

A: Hybrid search combines vector semantic search with BM25 keyword search, fusing results via Reciprocal Rank Fusion. Pure vector search can miss exact matches for specific terms (codes, names, IDs). Pure keyword search misses semantically similar content. Hybrid gets the benefits of both: high recall and semantic understanding, especially important for heterogeneous enterprise document corpora.

Q: How does incremental ingestion work, and why does it matter at scale?

A: Incremental ingestion tracks a "last run" timestamp and only processes blobs modified after that point. This avoids reindexing the entire document corpus every week, reducing Azure OpenAI embedding costs, Document Intelligence API calls, and processing time — critical at scale where a full corpus may contain thousands of documents.

Q: What chunking strategy is used and why?

A: We use a fixed-size sliding window (500 words, 50 word overlap). The overlap ensures that content at chunk boundaries isn't lost — a sentence that spans two chunk boundaries will be partially present in both, preserving coherence. Tables are chunked separately to preserve their tabular structure.

ADVANCED Questions

Q: Scenario: A user asks "Compare Q3 revenue in our financial reports with the industry average from the web." How does the agent handle this?

A: The agent receives the query and enters the ReAct loop. Step 1: It calls document_search("Q3 revenue financial reports") to retrieve internal report chunks. Step 2: It calls web_search("industry average Q3 revenue 2024") to get current web data. Step 3: With both results in context, the LLM generates a grounded comparative answer citing both sources. Both citation sets are returned to the user.

Q: How do you prevent hallucinations in the agentic RAG system?

A: Multiple guardrails: (1) The system prompt explicitly instructs the LLM to only use retrieved context and cite sources. (2) Retrieved chunks are passed as grounding context, making it harder for the LLM to deviate. (3) Tool outputs contain actual data, so the LLM must synthesize from real evidence. (4) Citations are returned alongside the answer, allowing users to verify. (5) Max reasoning steps prevents recursive tool loops that could generate speculative chains.

Q: How is the chat thread designed for context carry-over?

A: Each thread has a unique GUID. Messages are stored in order. On each agent call, the last N messages (configurable via MaxChatHistoryMessages) are injected into the LLM message history before the new user query. This allows follow-up questions like "What did you just say about revenue?" to work correctly. Thread reset clears messages but keeps the ID, so the user can start fresh without breaking external references.

Q: How would you scale this system to millions of documents?

A: (1) Azure AI Search scales with partitions and replicas — add partitions to increase index capacity. (2) Switch from in-memory cache to distributed cache (Redis) for multi-instance API scaling. (3) Use Azure Durable Functions for the ingestion pipeline to handle parallelism and retries. (4) Switch MemoryService from in-memory dictionary to Azure Cosmos DB for persistent, distributed user memory. (5) Use Azure Event Grid to trigger ingestion asynchronously when new blobs arrive, rather than weekly polling.

Q: What are the cost optimization strategies in this design?

A: (1) Batch embedding generation reduces OpenAI API calls per ingestion job. (2) In-memory caching avoids re-running LLM calls for identical queries. (3) Incremental ingestion avoids re-embedding unchanged documents. (4) Serverless SQL (auto-pause) reduces database cost when idle. (5) Standard_LRS storage (lower redundancy) for document source (documents can be re-ingested if lost). (6) Max reasoning steps cap prevents runaway agent loops that consume tokens.
