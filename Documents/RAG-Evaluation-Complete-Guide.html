1. Why Evaluate RAG?

A RAG system that "works" is not the same as one that's reliable. Without evaluation:

  • You won't know if it's hallucinating until a user complains
  • You can't measure if a prompt change made things better or worse
  • You can't block bad deployments before they reach production
🎯 Interview Answer: "Every production RAG needs automated evaluation with pass/fail thresholds. I evaluate retrieval and generation separately, run it on every PR, and block merges if faithfulness drops below 0.80."

2. Two Layers to Evaluate

A RAG pipeline is two systems — both can fail independently:

User Question │ ▼ ┌──────────────┐ ┌──────────────┐ │ RETRIEVER │────▶│ GENERATOR │ │ (Search) │ │ (LLM) │ └──────┬───────┘ └──────┬───────┘ │ │ Did it find Did it produce the RIGHT a GROUNDED chunks? answer? │ │ Precision@K Faithfulness Recall@K Groundedness MRR Relevance
If retrieval bad + generation goodConfident-sounding wrong answers (worst case)
If retrieval good + generation badGarbled version of correct information
Both goodTrustworthy answer
Key insight: Retrieval metrics need NO LLM calls (free, instant). Generation metrics use an LLM-as-judge (~$0.005 per eval). Always check retrieval first — if chunks are wrong, no amount of prompt tuning helps.

3. Retrieval Metrics

Pure math — no LLM call, no cost, deterministic. You need a golden test set that labels which chunks are relevant for each question.

3.1 Precision@K — "Of what I retrieved, how much is relevant?"

Formularelevant_in_top_k / k
ExampleRetrieved 5 chunks, 3 are relevant → 3/5 = 0.60
Good> 0.70

Worked example:

Question: "What is the termination notice period?"

Retrieved chunks (K=5):
  [1] "Section 12.1: Either party may terminate with 90 days written notice." → ✅ Relevant
  [2] "Appendix C: Fee schedule for 2024-2025."                              → ❌ Irrelevant
  [3] "Section 12.2: Notice must be delivered to the registered address."    → ✅ Relevant
  [4] "Section 1.1: This agreement is between Company A and Company B."      → ❌ Irrelevant
  [5] "Section 12.3: Exceptions to notice for material breach."              → ✅ Relevant

Precision@5 = 3 relevant / 5 total = 0.60

What this tells you: 40% of the context the LLM receives is noise.
The LLM wastes tokens processing fee schedules and party names.

Low precision means: LLM gets noisy context → worse answers + higher token cost. At K=10 with precision 0.30, 70% of your context window is wasted on junk.
Fix: Raise score threshold, enable semantic reranker (Azure AI Search built-in), add metadata filters (e.g., filter by document section type).

3.2 Recall@K — "Of all relevant chunks, how many did I find?"

Formularelevant_retrieved / total_relevant_in_corpus
Example8 relevant chunks exist, retrieved 3 → 3/8 = 0.375
Good> 0.60

Worked example:

Question: "What is the termination notice period?"

All relevant chunks in the entire corpus (ground truth):
  [A] "Section 12.1: Either party may terminate with 90 days written notice."
  [B] "Section 12.2: Notice must be delivered to the registered address."
  [C] "Section 12.3: Exceptions to notice for material breach."
  [D] "Section 12.4: Partial termination requires 30 days notice."

Our retriever found: [A], [B], [C] — missed [D]

Recall@5 = 3 found / 4 total = 0.75

What this tells you: The answer will miss "partial termination" info.
If a user asks about partial termination, they get an incomplete answer.

Low recall means: Missing important info → incomplete answers. Users get partial truths.
Fix: Increase top-K (from 5 to 8), improve embedding model, add synonym expansion, enable hybrid search (BM25 catches keyword matches that embeddings miss).

Precision vs Recall trade-off: Increasing K improves recall but may hurt precision. At K=5: clean but possibly incomplete. At K=15: comprehensive but full of noise. Sweet spot is usually K=5 for focused questions, K=8-10 for broad ones. A semantic reranker (Step 2 after retrieval) is the best fix — it lets you retrieve K=20 then rerank to the top 5, getting both high recall AND high precision.

3.3 MRR — "Is the best chunk ranked first?"

Formula1 / rank_of_first_relevant_result
ExampleFirst relevant at rank 1 → MRR = 1.0. At rank 3 → MRR = 0.33
Good> 0.80

Why MRR matters for RAG: LLMs have a "lost in the middle" bias — they pay most attention to the first and last chunks. If the relevant chunk is buried at position 4, the LLM anchors on irrelevant content above it.

🎯 Interview Answer: "MRR tells me if the retriever is front-loading relevant results. Even with high recall, if MRR is low, the LLM sees junk first — that's the 'lost in the middle' problem."

4. Generation Metrics

These measure: "Given the retrieved context, did the LLM produce a good answer?" They require an LLM-as-judge call (explained in section 5).

4.1 Groundedness — "Does the answer ONLY use info from context?"

The #1 metric. Non-negotiable.

What it checksEvery claim in the answer must be traceable to the retrieved context
Good> 0.90
Block deploys if< 0.80
Context: "The contract requires 90 days written notice for termination."

✅ Grounded: "The notice period is 90 days, in writing."
   → Score 1.0

❌ Hallucinated: "Notice is 90 days, sent via certified mail to the registered agent."
   → Score 0.5 ("certified mail" and "registered agent" invented)
🎯 Interview Answer: "Groundedness is my top priority. A grounded-but-incomplete answer is safe — the user asks a follow-up. A hallucinated-but-complete answer is dangerous — the user trusts wrong info. I block deployment if groundedness < 0.80."

4.2 Faithfulness — "Are the specific facts correct?"

Stricter than groundedness — checks individual facts (numbers, names, dates) against context.

TypeGroundedness FailureFaithfulness Failure
DefinitionInventing info not in contextDistorting facts from context
Example"sent via certified mail" (context never mentions mail method)"$1.5M cap" (context says "$1M cap")
Which is worse?Both are dangerousFaithfulness failures are harder to catch because the topic IS in context — the detail is just wrong

Detailed example — faithfulness failure in action:

Context: "Section 5.3: Late payments shall accrue interest at 1.5% per month
          on the outstanding balance, capped at 18% annually."

✅ Faithful: "Late payments incur 1.5% monthly interest, capped at 18% per year."
   → Score 1.0 (all facts match exactly)

❌ Unfaithful: "Late payments incur 2% monthly interest with no annual cap."
   → Score 0.0 (1.5% → 2%, 18% cap → "no cap" — both facts distorted)

❌ Subtly unfaithful: "Late payments incur 1.5% monthly interest, capped at 15% annually."
   → Score 0.5 (interest rate correct, annual cap wrong: 18% → 15%)
   → This is the HARDEST failure to catch — the answer looks right at a glance
Common LLM mistake: paraphrasing "90 days" as "3 months" — technically equivalent but NOT what the document says. Finance and legal teams care about exact wording. In a contract review RAG, this could be a compliance violation. That's why RAGAS decomposes into exact claims rather than holistic scoring.

4.3 Relevance — "Does it answer the actual question?"

What it checksAnswer directly addresses what was asked
Good> 0.80
Common failureAsk about termination → get answer about renewal (both in same document section)

Detailed example — relevance failure:

Question: "What is the termination notice period?"

Context includes both:
  - "Section 12: Termination requires 90 days notice."
  - "Section 11: Renewal requires 60 days notice."

❌ Low relevance answer: "The contract renewal requires 60 days written notice
   prior to the end of the current term."
   → Faithfulness: 1.0 (fact IS in context)
   → Relevance: 0.30 (user asked about TERMINATION, got RENEWAL)

This is why you need BOTH metrics. An answer can be perfectly faithful
to the context but completely miss the user's question.
🎯 Interview Answer: "Faithfulness and relevance catch different failures. An answer can be 100% faithful — every fact comes from context — but 0% relevant because it's answering the wrong question. Both happen when the retriever returns chunks about the right document but the wrong section. You need both metrics to trust the system."

5. LLM-as-Judge — How Every Eval Tool Works Inside

This is the core pattern used by RAGAS, DeepEval, Azure AI Foundry. Understanding this makes you credible in interviews — you know what's under the hood.

The pattern:

  1. Take the question, context, and generated answer
  2. Send them to a separate GPT-4o instance (not the one that generated the answer)
  3. Ask it to score groundedness, faithfulness, relevance on a 0-1 scale
  4. Use temperature = 0 for deterministic scoring
  5. Parse the JSON scores
# What RAGAS / DeepEval do internally (simplified):

Judge Prompt:
  "You are a strict evaluator. Score 0.0 to 1.0.
   Question: {question}
   Context: {retrieved_chunks}
   Answer: {generated_answer}

   groundedness: Does answer ONLY use info from context?
   faithfulness: Are specific facts correct per context?
   relevance: Does it answer the question?

   JSON only: {groundedness: 0.0, faithfulness: 0.0, relevance: 0.0}"

→ GPT-4o (temperature=0)
→ {"groundedness": 0.92, "faithfulness": 0.88, "relevance": 0.95}
ApproachCost per evalCorrelation with humans
Human reviewer~$2.001.0 (baseline)
GPT-4o as judge~$0.005~0.85
Rule-based (regex)$0~0.40
🎯 Interview Answer: "All major eval tools — RAGAS, DeepEval, Azure AI Foundry — use the same pattern: a separate GPT-4o instance judges each answer. Temperature 0 for reproducibility. ~$0.005 per eval, ~85% correlation with human reviewers."

6. Golden Test Set — The Only Manual Step

This is the most important step and the one teams skip. Without it, you're evaluating nothing.

What is it?

20-50 real questions with expected key-point answers, categorized by topic.

Rules:

  1. Real questions — from chat logs and support tickets, NOT invented by engineers
  2. Key points, not exact answers — list the facts the answer MUST contain
  3. Categories — so you can see which document types are failing
  4. Include adversarial questions (20%) — answer is NOT in the documents

Example:

[
  {
    "question": "What is the termination notice period?",
    "expected_key_points": "90 days, written notice, both parties",
    "category": "contracts",
    "answer_in_docs": true
  },
  {
    "question": "What is the CEO's personal phone number?",
    "expected_key_points": "Should refuse — not in documents",
    "category": "adversarial",
    "answer_in_docs": false
  }
]
Adversarial questions test hallucination resistance. A system that always answers — even when it shouldn't — is dangerous. 20% of your test set should be things the system should say "I don't know" to.
🎯 Interview Answer: "I build a golden test set of 30+ real questions from chat logs, with expected key points and categories. 20% are adversarial — the answer doesn't exist in our documents. This tests if the system says 'I don't know' vs. hallucinating."

7. RAGAS — Deep Dive

In production, you don't write custom evaluation code. You use RAGAS (Retrieval-Augmented Generation Assessment) — the most popular open-source RAG eval framework. Every concept from sections 3-6 maps directly to a RAGAS metric. This section shows exactly how.

7.1 How RAGAS Works Inside — Metric by Metric

RAGAS needs four columns of data. Each column feeds specific metrics:

Your RAG Pipeline produces: RAGAS evaluates: ───────────────────────── ──────────────────── question ─────────────────────▶ answer_relevancy (is answer on-topic?) context_precision (are retrieved chunks relevant?) context_recall (did we find all relevant chunks?) contexts ─────────────────────▶ context_precision (is junk mixed in?) (retrieved chunks from search) faithfulness (does answer stick to these chunks?) answer ─────────────────────▶ faithfulness (any hallucinated claims?) (LLM-generated response) answer_relevancy (does it address the question?) ground_truth ─────────────────────▶ context_recall (did retriever find chunks matching this?) (expected key points) answer_correctness (optional: is answer factually right?)

How each metric calculates — step by step:

faithfulness — Our Groundedness + Faithfulness combined

This is the most important metric. RAGAS does NOT score it as a single number — it decomposes the answer into individual claims, then checks each one.

RAGAS faithfulness — internal steps:

Step 1: DECOMPOSE answer into atomic claims
  Answer: "The notice period is 90 days. Notice must be written.
           It should be sent via certified mail."
  → Claims: ["notice period is 90 days",
             "notice must be written",
             "should be sent via certified mail"]

Step 2: CHECK each claim against context (LLM-as-judge)
  Claim 1: "notice period is 90 days" → Context says "90 days" → ✅ SUPPORTED
  Claim 2: "notice must be written"   → Context says "written notice" → ✅ SUPPORTED
  Claim 3: "sent via certified mail"  → Context says nothing about mail method → ❌ NOT SUPPORTED

Step 3: CALCULATE
  faithfulness = supported_claims / total_claims = 2/3 = 0.667
Why decomposition matters: If RAGAS just asked "is this answer faithful?" it would get vague 0.7-0.8 scores. By checking each claim separately, you can see exactly which sentence is hallucinated. This is actionable.

context_precision — Our Precision@K

RAGAS checks: of the chunks you retrieved, how many were actually useful for answering this question?

Step 1: Judge ranks each retrieved chunk as relevant or irrelevant
  Chunk 1: "Contract termination requires 90 days..." → ✅ Relevant
  Chunk 2: "Appendix A: pricing schedule..."          → ❌ Irrelevant
  Chunk 3: "Both parties must provide written..."     → ✅ Relevant
  Chunk 4: "Office hours are 9am to 5pm..."           → ❌ Irrelevant
  Chunk 5: "Notice period exceptions for breach..."   → ✅ Relevant

Step 2: context_precision = relevant / total = 3/5 = 0.60

context_recall — Our Recall@K

Uses ground_truth — did the retriever find chunks covering all expected key points?

ground_truth: "90 days, written notice, both parties, exceptions for breach"

Step 1: Break ground_truth into key points
  → ["90 days", "written notice", "both parties", "exceptions for breach"]

Step 2: Check if retrieved chunks contain each key point
  "90 days"              → Found in Chunk 1 ✅
  "written notice"       → Found in Chunk 3 ✅
  "both parties"         → Found in Chunk 3 ✅
  "exceptions for breach"→ Found in Chunk 5 ✅

Step 3: context_recall = found / total = 4/4 = 1.00

answer_relevancy — Our Relevance

RAGAS uses a reverse technique: it generates hypothetical questions from the answer, then checks similarity to the original question.

Original question: "What is the termination notice period?"

Step 1: Generate 3 hypothetical questions from the answer
  "What is the notice period for contract termination?"
  "How many days notice is required?"
  "What form must termination notice take?"

Step 2: Calculate embedding similarity between original and each generated question
  Similarity: 0.95, 0.88, 0.72

Step 3: answer_relevancy = average = 0.85
Why reverse? If the answer talks about the right topic, the generated questions will be similar to the original. If the answer drifts off-topic (e.g., discusses renewal instead of termination), the generated questions will be about renewal — low similarity to the original.

7.2 Full Worked Example — 3 Test Questions, Start to Finish

You prepare a golden_test_set.json with questions, expected answers, retrieved contexts, and ground truth — then call evaluate() with your chosen metrics and a judge LLM (GPT-4o at temperature 0). RAGAS scores every question individually.

7.3 RAGAS Output — What You Actually See

Overall scores (what print(result) shows):

{
  'faithfulness':       0.889,
  'answer_relevancy':   0.912,
  'context_precision':  0.722,
  'context_recall':     0.917
}

Per-question breakdown (what result.to_pandas() shows):

result_df = result.to_pandas()
print(result_df.to_string())

                                          question  faithfulness  answer_relevancy  context_precision  context_recall
0            What is the termination notice period?         1.000             0.950              0.667           1.000
1                    What are the payment terms?            1.000             0.920              1.000           1.000

How to read this output — question by question:

#QuestionScoreWhat Happened
Q1 Termination notice faithfulness: 1.0 All claims in the answer trace back to context. Perfect.
Q1 context_precision: 0.667 Retrieved 3 chunks, but "Appendix C: Fee schedule" was irrelevant → 2/3 useful
Q2 Payment terms All metrics > 0.90 Clean retrieval (2/2 relevant chunks) + clean generation. Ideal.
Q3 Contract transfer faithfulness: 0.667 Answer has 3 claims. 2 are supported. 1 is hallucinated — RAGAS flagged it.
Q3 context_recall: 0.75 Ground truth has 3 key points; retriever only found chunks covering 2 of them.
This is why per-question scores matter. The overall faithfulness (0.889) looks fine. But Q3 is 0.667 — that's a hallucination hiding in the average. Always check the per-question breakdown.

At the end of run_ragas.py, compare each metric against thresholds and sys.exit(1) if faithfulness or relevance fall below 0.80. This integrates directly into CI/CD to block merges.

8. CI/CD Quality Gate

Run evaluation automatically on every PR that touches prompts, search config, or chunking logic. A GitHub Actions workflow installs RAGAS, runs python eval/run_ragas.py, and fails the PR if thresholds are not met.

Thresholds that block a PR:

MetricBlock if belowWhy
Faithfulness0.80Hallucination risk — safety critical
Relevance0.80Answering wrong questions
Context Precision0.70Retrieval noise → worse answers
Rule: Block on safety metrics (faithfulness, groundedness). Warn on quality metrics (precision, recall). Never let hallucination regressions ship.

9. Production Monitoring — Continuous Evaluation

You can't run full LLM-judge on every live query (too slow at ~3 sec per eval, too expensive at scale). The real-world approach is a tiered monitoring system — cheap + fast checks on everything, expensive + thorough checks on a sample.

9.1 Four Monitoring Tiers

TierCoverageWhatCostLatencyCatches
1. Thumbs up/down 100% (opt-in by user) Button in chat UI $0 0ms User dissatisfaction — broad signal, easy to collect
2. Reflection score 100% of queries LLM self-check after generation ~$0.001/query ~500ms added Obvious hallucinations, incomplete answers, off-topic drift
3. RAGAS async eval 3-5% random sample Background job, fires after response sent ~$0.012/eval 0ms (async) Faithfulness drift, precision degradation, recall gaps
4. Human review 0.5-1% sample Weekly reviewer queue ~$2.00/review 24-48h Tone, clarity, nuance, edge cases automated tools miss
Live Query Flow with Monitoring: User Question │ ▼ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ Retrieve │───▶│ Generate │───▶│ Reflect │──▶ Response to user │ chunks │ │ answer │ │ (Tier 2) │ └─────┬──────┘ └─────┬──────┘ └─────┬──────┘ │ │ │ │ Log everything (Tier 1 ready) │ ▼ ▼ ▼ ┌──────────────────────────────────────────────┐ │ Request Log (App Insights / Cosmos DB) │ │ question | contexts | answer | reflection │ │ score | thumbs_vote | latency | timestamp │ └──────────────────┬───────────────────────────┘ │ ┌────────┴─────────┐ ▼ ▼ 3-5% sampled 0.5% sampled ┌──────────┐ ┌──────────┐ │ RAGAS │ │ Human │ │ Async │ │ Review │ │ (Tier 3) │ │ (Tier 4) │ └──────────┘ └──────────┘
Launch week strategy: Start Tier 3 at 10% sampling to establish baselines. After 1 week (once you have ~500+ evaluated queries), drop to 3-5%. Tier 4 human reviews: start immediately — 20-30 answers reviewed by a domain expert in week 1 to calibrate your expectations.

9.2 Alerting Rules

SignalSourceAlert IfLikely CauseSeverity
Weekly avg faithfulness Tier 3 RAGAS Drops below 0.85 Prompt regression or model update Critical — page on-call
Thumbs-down rate Tier 1 UI Exceeds 15% (7-day rolling) Retrieval or generation degradation Critical
"I don't know" rate drops suddenly Response parsing Week-over-week drop > 20% System started hallucinating instead of refusing Critical — most dangerous signal
"I don't know" rate spikes suddenly Response parsing Week-over-week spike > 30% Retrieval broke — returning empty results Warning

The scariest alert: "I don't know" rate dropping. It means the system stopped being cautious and started confidently answering things it shouldn't. This is the hallucination indicator you'll miss if you only watch faithfulness scores (because RAGAS only evaluates a 5% sample — the hallucinated answers might be in the other 95%).

9.4 Drift Detection — When Things Silently Break

The three types of drift that silently destroy RAG quality:

Drift TypeWhat HappensHow to DetectFix
Model drift Azure updates GPT-4o version silently. New version behaves differently. Log model_version per request. Compare faithfulness before/after version change. Pin model version in deployment: "gpt-4o-2024-08-06" not "gpt-4o". Test new versions before switching.
Data drift New documents indexed (quarterly reports, policy updates). Chunking may be wrong for new format. Track context_precision per document source. New sources will show lower precision. Test chunking on each new document type. Create category-specific golden test questions.
Query drift Users start asking different types of questions than what you trained/tested for. Cluster questions weekly using embeddings. Look for new clusters not covered by golden test set. Add new real questions to golden test set monthly. Expand categories.

Model version management — the #1 production lesson:

// ❌ BAD — Azure can change this anytime
var deployment = "gpt-4o";  // Points to "latest" — could change overnight

// ✅ GOOD — pinned to specific version
var deployment = "gpt-4o-2024-08-06";  // Specific version you tested
// When Azure releases new version:
//   1. Deploy to staging with new version
//   2. Run RAGAS golden test set against staging
//   3. Compare scores to baseline
//   4. Only switch production if scores >= baseline
🎯 Interview Answer: "In production, I monitor RAG at four tiers: thumbs-up/down on all queries, reflection scoring on 100%, RAGAS on a 5% async sample via Azure Functions, and weekly human review of 20-30 answers. I log question + contexts + answer + model version per request to Cosmos DB — that's the data RAGAS consumes async. The most dangerous drift is model version changes — Azure can update silently. I pin model versions and gate upgrades behind RAGAS evaluation. The scariest alert is 'I don't know' rate dropping — that means the system started hallucinating instead of refusing."

10. Diagnosing Failures — When Scores Drop

10.1 Failure Decision Tree

Bad Answer Reported │ ├─ Is Context Precision low? 11. Interview Deep Dive

11.1 The One-Paragraph Answer (memorize this)

"I evaluate RAG at two layers. Retrieval: did we find the right chunks? I measure Precision@K and MRR — no LLM needed, pure math against a golden test set. Generation: is the answer grounded? I use RAGAS with GPT-4o as an LLM judge — it decomposes every answer into atomic claims and checks each one against the context. I run this on 30+ real questions in CI/CD and block PRs if faithfulness drops below 0.80. In production, I log every request to Cosmos DB, run RAGAS async on a 5% random sample via Azure Functions, collect thumbs-up/down on all queries, and alert if faithfulness trends below 0.85 weekly."

11.2 Concept Questions — "Do You Understand Evaluation?"

Q: "How do you evaluate a RAG system?"
A: "Two separate layers — retrieval and generation — because they fail independently. Retrieval: did the search find the right chunks? I use Precision@K (how much retrieved content is relevant) and MRR (is the best chunk ranked first). Generation: is the answer grounded? I use RAGAS which scores faithfulness by decomposing the answer into individual claims and checking each against the context. The worst scenario is bad retrieval + good generation — the LLM sounds confident about wrong information."
Q: "What is faithfulness and why does it matter most?"
A: "Faithfulness = does the answer ONLY use facts from the provided context? Score 0 to 1. RAGAS calculates it by breaking the answer into atomic claims — 'notice period is 90 days', 'notice must be written', 'sent via certified mail' — then checks each claim against the chunks. If the context never mentions 'certified mail', that claim is unsupported → score drops. A grounded-but-incomplete answer is safe (user asks follow-up). A hallucinated-but-complete answer is dangerous (user trusts wrong info). That's why I block deployments below 0.80."
Q: "Explain Groundedness vs Faithfulness."
A: "Groundedness: is the claim IN the context at all? Faithfulness: is the claim ACCURATE per the context? Example: context says '$1M liability cap'. Answer says '$1.5M liability cap'. That passes groundedness (it's about liability caps, which is in context) but fails faithfulness (the number is wrong — $1M distorted to $1.5M). In RAGAS, the faithfulness metric covers both — it checks if each claim is supported AND accurate."
Q: "What's MRR and why does it matter for RAG specifically?"
A: "MRR = 1 divided by the rank of the first relevant result. If the best chunk is at position 1, MRR = 1.0. At position 4, MRR = 0.25. It matters for RAG because of the 'lost in the middle' problem — research shows LLMs pay most attention to the first and last chunks in context. If relevant info is buried at position 3 out of 5, the LLM anchors on the irrelevant chunks above it. High recall but low MRR means you found the right content but presented it in the wrong order."
Q: "What's a golden test set and how do you build one?"
A: "30-50 real questions from chat logs and support tickets — NOT invented by engineers. Each has expected key points (not exact answers) and a category (contracts, HR, finance). 20% are adversarial — questions where the answer is NOT in the documents. The adversarial set tests hallucination resistance: does the system say 'I don't have that information' or does it make something up? I update the golden set monthly by adding new real-user questions from categories where RAGAS scores are lowest."
Q: "How does LLM-as-judge work?"
A: "Every eval tool — RAGAS, DeepEval, Azure AI Foundry — uses the same pattern: send the question + context + answer to a separate GPT-4o instance (not the one that generated the answer) at temperature 0 for deterministic scoring. It scores groundedness, faithfulness, relevance on 0-1 scale. Cost: ~$0.005 per evaluation. Correlation with human reviewers: ~85%. It's not perfect — that's why you also do human review on 1% of traffic — but it's 400x cheaper and 100x faster than humans for the other 99%."

11.3 Scenario-Based Questions — "What Would You Do?"

Q: "Your faithfulness score dropped from 0.92 to 0.71 after a deployment. Walk me through your investigation."
A: "First, I check the per-question RAGAS breakdown — which specific questions failed? If all questions degraded: likely a prompt or model change. If only one category dropped: likely a retrieval issue with that document type.

1. Check model version — did Azure silently update GPT-4o? Compare model_version field in request logs before and after. This is the #1 surprise failure.
2. Check the prompt diff — did someone change the system prompt? Especially look for removed constraints like 'only answer from context'.
3. Check retrieval — pull the retrieved chunks for the failing questions. Are they relevant? If context_precision also dropped → retrieval problem, not generation.
4. Check chunking — were new documents indexed? New PDFs sometimes chunk differently → search returns garbage.
5. Roll back — if it's a code change, revert and confirm scores recover. If it's a model update, pin the previous version."
Q: "You're launching a new RAG system next week. Walk me through your evaluation plan."
A: "Three phases:

Phase 1 — Before launch (this week):
- Build golden test set: 30 real questions from stakeholders and support tickets. 20% adversarial.

Q: "Users are complaining about wrong answers but your RAGAS scores look fine. What's happening?"
A: "Three possible causes:

1. RAGAS is only evaluating 5% of traffic. The bad answers might be in the other 95%. Fix: temporarily increase sampling to 20% or pull the specific complaint questions and run RAGAS manually.
2. Relevance problem, not faithfulness. The answer is grounded (facts come from context) but it's answering the wrong question. The user asked about termination but got information about renewal — both topics exist in the same document. Check answer_relevancy scores by category.
3. Tone/format problem. The answer is correct but too long, too technical, or presented poorly. RAGAS doesn't measure this. Fix: human review (Tier 4) on the specific complaint patterns."
Q: "How do you evaluate multi-turn conversations?"
A: "Multi-turn is harder because context carries over. I evaluate at two levels:
- Per-turn: same RAGAS evaluation on each individual turn (question + context + answer).
- Conversation-level: did the overall conversation resolve the user's intent? I check: does the last answer contain all key points from the golden test set? I also check that the system doesn't contradict itself between turns — e.g., saying '90 days' in turn 1 and '60 days' in turn 3. For the golden test set, I include 5-10 multi-turn scenarios with expected resolution across 3-4 turns."
Q: "All metrics dropped at the same time but nobody changed the code. What happened?"
A: "Three likely causes in order of probability:
1. Azure updated the model version — this is the #1 surprise in production. Check the model_version field in your request logs. If it changed, pin the previous version and test the new one separately.
2. Search index corrupted or rebuilt — an index rebuild can change document IDs and scores. Check if index stats (document count, last updated timestamp) changed.
3. Embedding model drift — if you upgraded the embedding model without re-indexing all documents, old and new embeddings are incompatible. Similarity scores become meaningless. Fix: re-index all documents with the new embeddings."

12. Real-World Hallucination Detection & Resolution

Hallucination is the #1 trust-killer in production RAG. A system that confidently invents facts is worse than one that says "I don't know." This section covers the full pipeline: detect → classify → resolve → prevent.

12.1 Taxonomy of Hallucinations

Not all hallucinations are equal. Understanding the type tells you where the fix belongs:

TypeDescriptionExampleRoot CauseFix Layer
Intrinsic Contradicts the retrieved context Doc says "30-day refund" → answer says "60-day refund" LLM training data overrides context Prompt + Reflection
Extrinsic Adds facts not in any retrieved chunk Answer mentions a product feature that no document describes LLM fills gaps from parametric memory Prompt + Empty-result guard
Fabricated Citation Cites [DocSource 3] but claim isn't in source 3 "Per [DocSource 2], SLA is 99.9%" — but source 2 is about billing LLM associates citation markers loosely Citation verification
Numeric Drift Changes numbers, dates, percentages Doc says $14,500 → answer says $15,000 (rounded/changed) LLM approximates numbers Extraction + comparison
Confident Empty Answers when tool results were empty Retriever found nothing → answer invents plausible facts No guardrail on empty context Empty-result guard
Real-world stat: In production RAG systems, extrinsic hallucinations (adding unsupported facts) account for ~60% of cases. Intrinsic (contradicting sources) is ~25%. Fabricated citations ~15%. Fixing extrinsic alone cuts total hallucination by more than half.

12.2 Detection Pipeline

A multi-layered approach — no single technique catches everything:

Answer Generated │ ├─── Layer 1: EMPTY-RESULT GUARD (free, instant) │ All tools returned "no results"? │ └─ YES → force "I don't have enough information" response │ └─ NO → continue │ ├─── Layer 2: CLAIM DECOMPOSITION (LLM call ~$0.003) │ Break answer into atomic claims: │ "The SLA is 99.9% and includes 24/7 support" │ → Claim 1: "The SLA is 99.9%" │ → Claim 2: "The SLA includes 24/7 support" │ ├─── Layer 3: CLAIM VERIFICATION (LLM call ~$0.005) │ For each claim, check against retrieved chunks: │ → Claim 1: SUPPORTED (DocSource 2, line 14) │ → Claim 2: NOT SUPPORTED (no chunk mentions 24/7 support) │ ├─── Layer 4: CITATION CROSS-CHECK (regex + LLM ~$0.002) │ [DocSource N] claims → verify fact exists in source N │ ├─── Layer 5: NUMERIC EXTRACTION (regex, free) │ Extract numbers from answer vs numbers from context │ Flag any number in answer not found in tool results │ └─── VERDICT - All claims supported → ✅ PASS - 1+ unsupported claims → ⚠️ HALLUCINATION DETECTED → Trigger resolution (retry with grounding prompt)
🎯 Interview Answer: "I use a 5-layer hallucination pipeline: empty-result guard catches the obvious case; claim decomposition breaks complex answers into verifiable statements; claim verification checks each against retrieved chunks using RAGAS faithfulness decomposition; citation cross-check validates [DocSource N] references; numeric extraction flags changed numbers. Layers 1 and 5 are free — no LLM cost. The full pipeline adds ~$0.01 per answer."

The enhanced ReflectionService passes actual tool results to the evaluator (not just the answer), enabling it to cross-check specific claims. A structured ReflectionResult returns sub-scores for groundedness, completeness, citations, and clarity, plus a boolean HallucinationDetected flag. An empty-result guard in the orchestrator short-circuits with "I don't have enough information" whenever all tools returned no data — preventing the LLM from fabricating.

12.3 Resolution Strategies

When hallucination is detected, the fix depends on the type:

Detection ResultResolution StrategyImplementation
HallucinationDetected = true Grounding-focused retry Retry prompt: "Your answer contained unsupported claims. Regenerate using ONLY facts from tool results. Remove anything you cannot cite."
GroundedScore = 1 (severe) Strip and regenerate Discard answer completely. Re-prompt with explicit: "List only facts from the following context. If a fact isn't below, don't mention it."
Fabricated citation Citation stripping Regex: find [DocSource N] → verify claim exists in source N text → remove unverifiable citations
Numeric drift Number anchoring Extract all numbers from tool results. Replace any answer number not in the set with "[exact figure in document: X]"
Confident empty Hard block Empty-result guard (Layer 1) — never reaches the LLM
Still hallucinating after retry Escalate to GPT-4o + strict template Switch to the more capable model with a fill-in-the-blank template: "Based on [source], the answer is [quote]."
Why this matters: The old approach used the same retry prompt for hallucination and incompleteness. This is counterproductive — telling a hallucinating model to "search for more information" just gives it more context to hallucinate around. The grounding-focused prompt specifically instructs it to remove unsupported claims.

12.4 Production Hallucination Monitoring

Detection in production requires a tiered approach (you can't run $0.01 checks on every request at 10K RPM):

TierCheckCoverageCostLatency
1 Empty-result guard 100% of requests $0 0ms (regex)
2 Reflection with tool results 100% of requests ~$0.003/req ~500ms
3 Full claim decomposition (RAGAS faithfulness) 5% sample + all low-scoring ~$0.01/req Async (not user-facing)
4 Human review of flagged answers 1% + all tier-3 failures $2-5/review Hours (batch)

Key metrics: Hallucination Rate (target <5%, alert >10%), "I Don't Know" Rate (target 5-15%, alert if drops below 3%), Grounded Score median (alert if <2.5/3), Citation Accuracy (target >95%), Retry Rate (target <10%, alert >20%).

🎯 Interview Answer: "The most dangerous metric to watch is the 'I don't know' rate dropping. A healthy RAG system should refuse 5-15% of questions. If that drops to 2%, it means the model started hallucinating instead of refusing — it's confidently making things up. I alert on both hallucination rate rising AND refusal rate falling."

14. Latency Optimization in RAG Systems

Users expect sub-second responses. A RAG pipeline that takes 8-12 seconds kills adoption. Latency hides in every layer — retrieval, generation, and the orchestration glue between them. This section covers the proven strategies to cut end-to-end latency without sacrificing quality.

14.1 Retrieval Layer

StrategyHow It WorksTypical SavingsTrade-off
Semantic caching Hash the embedding of the query; if a near-duplicate was answered recently, return the cached result Eliminates retrieval + generation entirely on cache hit (~95% reduction) Stale answers if source data changes — set a TTL (e.g., 1 hour)
Hybrid search + pre-filtering Add keyword pre-filter (date range, category) before the vector search so ANN scans a smaller index partition 30-50% faster retrieval on large indexes (>1M docs) Requires well-tagged metadata at ingestion time
Reduce top-K Retrieve 3 chunks instead of 10; fewer chunks = shorter prompt = faster generation ~200-400ms off generation (fewer input tokens) Risk of missing relevant context — validate with Recall@K on golden set
Quantized / binary vectors Use scalar or binary quantization on stored embeddings — reduces memory and speeds ANN search 2-4x faster search with <2% recall loss Slight recall degradation — use oversampling + re-rank to compensate

14.2 Generation Layer

StrategyHow It WorksTypical SavingsTrade-off
Streaming (SSE) Stream tokens to the UI as they're generated — perceived latency drops to first-token time (~200ms) Perceived wait goes from 3-5s to <500ms Total generation time unchanged — but UX feels instant
Complexity-based model routing Route simple queries (greetings, FAQs) to a smaller/faster model (GPT-4o-mini); reserve GPT-4o for complex multi-hop 50-70% faster on simple queries (majority of traffic) Needs a reliable complexity classifier — misrouting hurts quality
Shorter system prompts Trim verbose instructions; every 1K tokens adds ~100ms 100-300ms per 1-3K tokens removed Too short = less consistent output format
Temperature 0 Deterministic sampling skips the multinomial sampling step Small but measurable (~50ms) Less creative output — fine for factual RAG
