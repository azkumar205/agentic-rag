LLM Observability & Monitoring

Ground truth setup · RAGAS evaluation · LLM-as-judge · Human review · Cost & latency monitoring · OpenTelemetry · Interview revision

Overview

Observability for an LLM system answers three questions: Is the answer correct? (quality), How much did it cost? (cost), and How fast was it? (latency). Without observability, you cannot improve, debug, or trust your system in production.

In this Agentic RAG system, observability is implemented across 6 pillars:

Key insight: Observability is not a single tool — it's a stack: structured logging (ILogger) → custom metrics (OpenTelemetry) → quality evaluation (RAGAS + LLM-as-judge) → cost tracking (TokenUsageInfo) → alerting (Azure Monitor). Each layer catches different failure modes.

Real-World Walkthrough — Follow One Question Through the System

The best way to understand observability is to trace a single real question through every layer. Here are three scenarios you'd see in production.

Scenario 1: Happy Path — User Asks "What is the contract termination notice period?"

This is a simple factual question. Watch how every observability layer fires.

T+0ms — Request arrives
Step 1: Controller Logging
AgentController receives POST /api/agent/ask. Logs first 100 chars of question (PII-safe truncation).
INFO: Received question (first 100 chars): "What is the contract termination notice period?" INFO: SessionId = abc-123-def
T+5ms — PII Check (Layer 1)
Step 2: PII Redaction on Input
PiiRedactionService scans the question. No PII found. Logs the count (not the content).
INFO: PII Layer 1: Redacted 0 entities from user input
T+8ms — Intent Classification
Step 3: Rule-Based Intent (Tier 1)
IntentClassifierService matches "what is" pattern → FactualLookup (confidence 0.80). Zero LLM cost, ~1ms.
INFO: Intent classified as FactualLookup (confidence=0.80) — keyword "what is" matched
T+10ms — Cache Check
Step 4: Semantic Cache Lookup
SemanticCacheService embeds the question (text-embedding-3-small, 512 dims). Searches Azure AI Search cache index. No match above 0.92 cosine threshold → CACHE MISS.
INFO: Semantic cache: MISS (best match cosine=0.78, threshold=0.92)
T+120ms — Memory Load
Step 5: Conversation Memory
ConversationMemoryService loads last 5 turns from Redis. First question in session → empty. Logs turn count.
INFO: Loaded 0 history turns for session abc-123-def
T+130ms — LLM Planning Call
Step 6: GPT-4o-mini Plans + Calls Tools
PromptTemplateService selects FactualLookup system prompt (with CoT steps + few-shot examples). GPT-4o-mini decides to call SearchDocumentsAsync. The response includes [INTENT: FactualLookup] — LLM confirms the rule-based intent (Tier 2 piggyback, zero extra cost).
INFO: Planning prompt selected: FactualLookup (CoT + 2 few-shot examples) INFO: Calling tool (via MCP): SearchDocumentsAsync("contract termination notice") INFO: MCP tool returned: 3 chunks (tokens≈280) INFO: LLM Intent: FactualLookup (confirms rule-based — no override needed)
T+950ms — Complexity Routing
Step 7: Route to Simple (GPT-4o-mini reuses planning response)
ComplexityRouter: 1 tool called + 280 tokens + "what is" prefix → Simple. No GPT-4o generation needed. Planning response IS the answer. Saves $0.015.
INFO: Routing: Simple (tools=1, tokens≈280) INFO: Reusing planning response as final answer (Simple path)
T+960ms — Strip Intent Tag
Step 8: Clean the Answer
StripIntentTag() removes "[INTENT: FactualLookup]\n" from the beginning of the response. User never sees the tag.
T+970ms — Quality Check
Step 9: ReflectionService (LLM-as-Judge)
ReflectionService.EvaluateAsync() calls GPT-4o-mini with the question + answer + tools used. Scores: GROUNDED=3, COMPLETE=2, CITED=2, CLEAR=1 → Score 8/10. Above threshold (6). No retry needed.
INFO: Reflection: Score 8/10 — passed (threshold=6, cost=$0.002)
T+1380ms — Cache Write + Response
Step 10: Cache Answer + Return to User
Answer cached in semantic cache (for next similar question). Memory updated in Redis. Full AgentResponse returned.
INFO: Cached answer for future similar questions (TTL=4h) INFO: Response complete — 1380ms, score=8/10, model=gpt-4o-mini, intent=FactualLookup

Scenario 2: Retry Path — LLM Forgets to Cite Sources (Score 4 → 8)

This is the real-world scenario interviewers love — show how the system self-corrects.

Steps 1–7: Same as Scenario 1
Everything up to generation works fine
The question is "Compare contract A and contract B termination clauses." Intent: ComparisonAnalysis. Route: Complex (GPT-4o). Generation produces a good comparison... but forgets to add [DocSource] citations.
T+3200ms — Quality FAIL
ReflectionService scores 4/10
GROUNDED=3, COMPLETE=1, CITED=0, CLEAR=0 → Score 4. Below threshold (6). Retry triggered.
WARN: Reflection score 4/10 — diagnosing failure (attempt 1/2)
T+3210ms — Diagnose Failure
DiagnoseFailure() identifies: MissingCitations
The orchestrator checks: tools were called ✓, results weren't empty ✓, no tool errors ✓, but answer doesn't contain "[DocSource" → diagnosis: MissingCitations.
WARN: Diagnosed failure type: MissingCitations WARN: Retry prompt: "Add [DocSource N] citations to every factual claim. Each claim MUST reference the specific document section."
T+3220ms — Targeted Retry
GPT-4o re-generates with correction prompt
The LLM receives the SAME tool results, but now with a specific correction: "Add [DocSource N] citations to every factual claim." This is NOT "try again" — it's a targeted fix for the exact failure mode.
INFO: Retry 1/2 — re-generating with correction: MissingCitations INFO: Correction prompt appended to system message
T+5500ms — Quality PASS
ReflectionService scores 8/10 on retry
GROUNDED=3, COMPLETE=2, CITED=2, CLEAR=1 → Score 8. Above threshold. Answer delivered to user.
INFO: Retry succeeded — score improved from 4 to 8 (failure was MissingCitations)

Why this matters: Without the reflection loop, the user would have received an uncited comparison. With it, the system caught the problem, diagnosed it as "missing citations" (not generic "bad answer"), and fixed it in one retry. Total extra cost: ~$0.017 (one retry). Total extra latency: ~2.3s. But the user got a properly cited answer.

Interview Key Point: "The key difference from naive retry is DIAGNOSIS. We don't say 'try harder' — we analyze WHY: was it missing citations? no tools called? SQL error? empty search results? Then we give the LLM a specific fix. We have 5 targeted failure modes. This fixes ~80% of issues on first retry vs ~40% for generic retry. And the failure type is logged — so we can track WHICH failure modes happen most often and fix the root cause in the prompt."

Scenario 3: Cache Hit — Same Question 30 Minutes Later

Shows the 99.9% cost reduction and 95% latency reduction in practice.

T+0ms — Request arrives
A different user asks: "What's the termination notice for the contract?"
Different wording, same meaning. This is why we use semantic cache, not exact-match cache.
T+120ms — Cache HIT
Cosine similarity = 0.96 (threshold 0.92) → HIT!
The embedding of "What's the termination notice for the contract?" is 0.96 similar to the cached "What is the contract termination notice period?" — above 0.92 threshold.
INFO: Semantic cache: HIT (cosine=0.96, threshold=0.92) INFO: Returning cached answer — skipping full pipeline
T+150ms — Response returned
Total time: 150ms (vs 1380ms). Cost: ~$0.000004 (vs $0.003)
The entire pipeline is skipped — no planning, no tools, no generation, no reflection. The previous answer was already scored 8/10, so the quality is known-good.

What the client receives on cache hit:

{
  "answer": "The contract termination notice period is 90 days written notice. [DocSource 1]...",
  "reflectionScore": 8,
  "fromCache": true,          // ← Key observability signal
  "modelUsed": "cached",
  "reasoningSteps": ["Semantic cache: HIT (cosine=0.96)"],
  "tokenUsage": { "estimatedCost": 0.000004, "toolCallCount": 0 }
}

Interview Key Point: "This is why semantic cache is game-changing for LLM costs. We use vector similarity (text-embedding-3-small, 512 dims) with a 0.92 cosine threshold. Different wording, same meaning → cache hit. The cost drops from $0.003 to $0.000004 — that's 99.9% reduction. But here's the important part: the `fromCache: true` flag in every response means our dashboard can track cache hit rate in real time. If hit rate drops below 20%, we get an alert — maybe the threshold is too strict, or the index got corrupted."

Pillar 1 — Pipeline Telemetry

Every step of the 10-step pipeline emits structured log events and appends to the ReasoningSteps list, which is returned in the API response for full client-side observability.

Reasoning Steps (in-band tracing)

Each pipeline stage appends a human-readable step to reasoningSteps. This list is included in every AgentResponse, giving the frontend a transparent view of what the agent did:

// From AgentOrchestrator.cs — each step adds an entry
reasoningSteps.Add($"Intent: {intentResult.Intent} (confidence={intentResult.Confidence:F2})");
reasoningSteps.Add($"Routing: {complexity} (tools={toolsUsed.Distinct().Count()}, tokens≈{contextTokenEstimate})");
reasoningSteps.Add($"Reflection: Score {reflectionScore}/10 — diagnosing failure...");
reasoningSteps.Add($"Fallback: detected tool errors, retrying with alternative approach");

The client receives something like:

{
  "reasoningSteps": [
    "Intent: DataRetrieval (confidence=0.85) — SQL-oriented keywords detected",
    "Calling tool (via MCP): GetSchemaAsync",
    "MCP tool returned results",
    "Calling tool (via MCP): QuerySqlAsync",
    "MCP tool returned results",
    "LLM Intent: DataRetrieval (confirms rule-based)",
    "Routing: Simple (tools=2, tokens≈340)",
    "Routing: intent override — ComparisonAnalysis escalated to Complex",
    "Generation: GPT-4o synthesized answer (intent-specific prompt: DataRetrieval)",
    "Reflection: Score 8/10"
  ]
}

Interview Key Point: "Reasoning steps are a form of in-band distributed tracing. Every API response includes the agent's full decision trail — which tools were called, which model was used, why, and what quality score was assigned. Zero extra infrastructure needed; the tracing is baked into the response contract."

Token & Cost Tracking

The TokenUsageInfo model is attached to every AgentResponse:

// AgentModels.cs
public class TokenUsageInfo
{
    public int PromptTokens { get; set; }       // Input tokens consumed
    public int CompletionTokens { get; set; }   // Output tokens generated
    public int TotalTokens { get; set; }        // Sum
    public decimal EstimatedCost { get; set; }  // USD at current Azure OpenAI pricing
    public int ToolCallCount { get; set; }      // How many MCP tools were called
}

Cost formula per request:

// EstimatedCost calculation (approximate Azure pricing, April 2026)
// GPT-4o-mini planning:  $0.15  / 1M input tokens + $0.60  / 1M output tokens
// GPT-4o  generation:    $2.50  / 1M input tokens + $10.00 / 1M output tokens
// text-embedding-3-small (cache): $0.020 / 1M tokens
// text-embedding-3-large (docs):  $0.130 / 1M tokens
// ReflectionService call:         ~$0.002 per evaluation (cheap GPT-4o-mini call)

EstimatedCost =
    (PromptTokens * planningRate) +
    (CompletionTokens * completionRate) +
    (embeddingTokens * embeddingRate) +
    reflectionCost;

Current gap: EstimatedCost and ToolCallCount are populated in the model, but token counts from the Azure OpenAI SDK response are not yet explicitly extracted and wired into the field. The field exists but may show 0. See the Gaps section for the fix.

Pillar 2 — LLM-as-Judge (ReflectionService)

LLM-as-judge uses a separate, independent LLM call to evaluate the quality of the answer before it is returned to the user. This is the in-production quality gate.

ReflectionService — How It Works

1
Generate answer — planning + generation phases produce the final answer.
2
Score it — ReflectionService.EvaluateAsync(question, answer, toolsUsed) calls GPT-4o-mini with a strict scoring prompt.
3
Gate it — if score < ReflectionThreshold (default 6), the orchestrator diagnoses the failure type and retries (max 2 retries).
4
Return it — the final score is included in AgentResponse.ReflectionScore for every request.
// ReflectionService.cs — the scoring prompt
new(ChatRole.System, """
    You are an answer quality evaluator. Score the answer from 1-10.
    Scoring criteria:
    - GROUNDED (1-3): Is the answer based on actual tool results, not made up?
    - COMPLETE (1-3): Does it address all parts of the question?
    - CITED (1-2): Are sources cited with [DocSource] or [SQLSource]?
    - CLEAR (1-2): Is the answer well-structured and easy to understand?
    Respond with ONLY a single integer 1-10. Nothing else.
""")

Scoring Criteria Breakdown

CriterionMax PointsWhat It ChecksDiagnostic Failure Mode
GROUNDED3Answer based on tool results, not hallucinationNoToolsCalled → retry with "you MUST call tools"
COMPLETE3All parts of the question addressedLowQuality → retry with "provide thorough answer"
CITED2Sources tagged [DocSource N] / [SQLSource]MissingCitations → retry with "add inline citations"
CLEAR2Well-structured, easy to understandLowQuality → retry with format guidance
Total10 points — threshold 6 (configurable)
 
// From DiagnoseFailure() in AgentOrchestrator.cs
// 5 targeted failure modes — NOT generic "try harder"
if (toolsUsed.Count == 0)
    return ("NoToolsCalled", "You MUST search before answering...");

if (!answer.Contains("[DocSource") && !answer.Contains("[SQLSource]"))
    return ("MissingCitations", "Add [DocSource N] citations to every claim...");

if (toolErrors.Any(e => e.ErrorType == "ToolFailure"))
    return ("ToolError", "Try GetSchemaAsync before SQL queries...");

if (toolErrors.Any(e => e.ErrorType == "EmptyResult"))
    return ("EmptyResults", "Try different search keywords or SearchWebAsync...");

return ("LowQuality", "Provide a more thorough, grounded answer...");

Interview Key Point: "Targeted retry is the key difference from naive 'try again'. We diagnose WHY the reflection score is low — no citations, no tools called, SQL error, empty results — and give the LLM a specific correction prompt. This fixes ~80% of issues on first retry vs ~40% for generic retry. The failure type is also logged and returned in the response for observability."

✅ Pros of LLM-as-Judge

  • Scales to millions of answers (no human bottleneck)
  • Consistent scoring criteria across all requests
  • In-line — blocks bad answers from reaching users
  • Cheap (~$0.002/eval with GPT-4o-mini)
  • Triggers targeted retry, not just re-generation

❌ Cons of LLM-as-Judge

  • Can be fooled by confident-sounding hallucination
  • Scoring not perfectly calibrated to human preference
  • Adds ~400ms latency per request
  • Judge and generator share same training distribution
  • Score of 6/10 ≠ absolute quality guarantee

Pillar 3 — Evaluation Framework

Ground Truth Setup

The golden test set (eval/golden_test_set.json) is the foundation of all offline evaluation. Each entry contains a question, the expected answer, and the relevant document chunks that should be retrieved.

// eval/golden_test_set.json — current structure
[
  {
    "question": "What is the contract termination notice period?",
    "expected_answer": "90 days written notice",
    "relevant_chunks": [
      "Section 12.1: Either party may terminate with 90 days written notice.",
      "Section 12.2: Notice must be delivered to the registered address."
    ]
  }
]

How to build a high-quality golden test set:

  1. Domain expert questions — Ask SMEs who know the documents to write realistic questions and verified answers.
  2. Adversarial questions — Include questions where the correct answer is "not found" to test hallucination resistance.
  3. Multi-hop questions — "Which vendor has the highest disputed invoice total?" (requires SQL + doc search).
  4. Temporal questions — "Which contracts expire this quarter?" (tests date reasoning).
  5. Negation questions — "Which contracts do NOT include an SLA?" (tests absence detection).
  6. Continuous growth — At minimum 50 questions for statistical significance. 200+ for production confidence.

Current gap: The golden test set has only 3 questions. This is too small for reliable quality measurement. Any random variation will appear significant. Expand to at least 50 questions before trusting the eval numbers.

RAGAS Metrics — What They Measure

RAGAS (Retrieval Augmented Generation Assessment) provides 4 metrics that together fully characterize RAG quality:

MetricWhat it measuresFormula conceptThresholdFailure means
Faithfulness Is every claim in the answer supported by the retrieved context? Claims supported by context / total claims ≥ 0.80 Hallucination — LLM added facts not in documents
Answer Relevancy Does the answer actually address the question? Cosine similarity: question ↔ answer embedding ≥ 0.70 Off-topic response or incomplete answer
Context Precision Are the retrieved chunks actually relevant? Relevant chunks retrieved / total chunks retrieved ≥ 0.70 Noisy retrieval — returning irrelevant chunks
Context Recall Were all relevant chunks found? Relevant chunks found / total relevant chunks in golden set ≥ 0.70 Missing important context — retrieval gap

The quadrant rule:
High Faithfulness + High Relevancy = good generation.
High Precision + High Recall = good retrieval.
You need all four. A system can score 1.0 faithfulness by just saying "I don't know" — which scores 0.0 relevancy. RAGAS catches this.

Confusion Metrics for RAG

Standard ML confusion metrics (precision/recall/F1) apply to RAG in two ways:

Retrieval Confusion Matrix

RetrievedNot Retrieved
Relevant chunkTrue Positive (TP)False Negative (FN) — recall failure
Irrelevant chunkFalse Positive (FP) — precision failureTrue Negative (TN)

Retrieval Precision = TP / (TP + FP) — are the retrieved chunks useful?
Retrieval Recall = TP / (TP + FN) — did we find all relevant chunks?
Retrieval F1 = 2 × (Precision × Recall) / (Precision + Recall) — harmonic mean

Answer Quality Classification

For factual questions with ground truth, you can classify each answer:

CategoryDefinitionAction
CorrectAnswer matches ground truth, well-citedCache it, use it for few-shot
Partially CorrectRight direction, missing detailsLog for prompt improvement
HallucinatedConfidently wrong or fabricatedImmediate alert, retrain retrieval
RefusedSaid "not found" when answer existsImprove retrieval coverage
Off-topicAnswered different question entirelyImprove intent classification

Evaluation Implementation

# eval/run_eval.py — current implementation
import json, requests, pandas as pd
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

# Load golden test set
with open("golden_test_set.json") as f:
    golden = json.load(f)

results = []
for item in golden:
    # Call the live API
    resp = requests.post(API_URL, json={"question": item["question"]})
    data = resp.json()

    results.append({
        "question": item["question"],
        "answer": data["answer"],
        "contexts": [c["content"] for c in data.get("textCitations", [])],
        "ground_truth": item["expected_answer"],
        "relevant_chunks": item["relevant_chunks"]
    })

# Run RAGAS evaluation
report = evaluate(pd.DataFrame(results),
                  metrics=[faithfulness, answer_relevancy,
                           context_precision, context_recall])

# Quality gates — CI/CD pipeline fails if any metric drops below threshold
assert report["faithfulness"]      >= 0.80, "Faithfulness below threshold"
assert report["context_precision"] >= 0.70, "Context precision below threshold"
assert report["context_recall"]    >= 0.70, "Context recall below threshold"
assert report["answer_relevancy"]  >= 0.70, "Answer relevancy below threshold"

Pros & Cons of This Observability Architecture

LLM-as-Judge (ReflectionService)

✅ Pros

  • Scales infinitely — one judge per request, no human needed
  • Consistent criteria across all answers
  • In-line blocking of bad answers before delivery
  • Dirt cheap (~$0.002 per eval)
  • Powers targeted retry — not "try harder" but "fix THIS specific thing"
  • Reflection score in API response = zero-extra-call quality signal for dashboards

❌ Cons

  • Same LLM family → same blindspots (can't catch what GPT-4o missed)
  • Adds ~400ms to every request
  • Judge score ≠ human preference (needs periodic calibration)
  • Adversarial answers can game the judge
  • GPT-4o-mini may inflate scores for confidently-wrong answers

RAGAS Offline Evaluation

✅ Pros

  • 4 metrics cover both retrieval and generation quality
  • CI/CD integration — breaks build on quality regression
  • Framework handles complex metric logic automatically
  • Reproducible — same golden set every run

❌ Cons

  • Requires ground truth (expensive to create)
  • Golden set can become stale if documents change
  • RAGAS itself uses LLM calls internally (costs money to run eval)
  • 3-question set is statistically meaningless — needs 50+

OpenTelemetry + Azure Monitor

✅ Pros

  • Free default telemetry (HTTP traces, dependencies, exceptions)
  • Custom metrics integrate via standard Meter API
  • KQL is very powerful for correlation and alerting
  • No vendor lock-in for the OTel instrumentation itself

❌ Cons

  • Custom metrics require additional code (not auto-extracted from AI SDK)
  • Azure Monitor pricing ($2.76/GB ingested) — noisy logging can be expensive
  • Dashboard setup requires up-front KQL knowledge
  • 15-min ingestion delay — not truly real-time

Semantic Cache Observability

✅ Pros

  • Cache hits produce FromCache: true in every response — zero-cost observability
  • 99.9% cost reduction per hit is measurable directly from EstimatedCost
  • Tunable threshold in config — can A/B test 0.90 vs 0.95

❌ Cons

  • Hit rate not currently exported as a metric — requires code addition
  • Stale cached answers not detectable without expiry + freshness metadata
  • High threshold (0.92) means semantically similar questions still miss

Interview Key Points — Last-Minute Revision

"What is LLM observability and why is it different from traditional software observability?"
Traditional observability tracks: did the code execute correctly? LLM observability tracks: was the output correct? You can have a 200 OK response with a hallucinated answer. LLM observability must evaluate the semantic quality of output, not just the mechanics of execution. This requires three new layers: LLM-as-judge (in-production quality scoring), offline evaluation against ground truth (RAGAS), and cost tracking (because wrong answers are expensive).

"How do you prevent hallucination in your system?"
Three mechanisms: (1) RAG itself — the agent must call tools before answering; the system prompt says "NEVER answer without tool results". (2) Reflection scoring — after generation, ReflectionService scores groundedness 1-3; a score of 1-2 on GROUNDED triggers a retry with "you MUST use tool results". (3) Citation gates — the CITED score (1-2) forces the LLM to tag every fact with [DocSource N] or [SQLSource]; uncited answers fail reflection and get retried.

"What is LLM-as-judge and what are its limitations?"
LLM-as-judge uses a separate LLM call (GPT-4o-mini) to evaluate the quality of answers on a 1-10 scale across 4 criteria: grounded, complete, cited, clear. The key advantage is scale — it evaluates every response, not just sampled ones. The key limitation is that the judge uses the same model family as the generator, so it can have the same blindspots. Also, LLMs can be misled by confident-sounding hallucinations. That's why we combine LLM-as-judge with human review (user thumbs-up/down) and offline RAGAS evaluation against ground truth.

"How do you set up ground truth / a golden test set?"
Ground truth is question-answer-context triples created by domain experts. Good golden sets include factual questions (has a definitive answer), adversarial questions ("not found" is correct), multi-hop questions (require combining two sources), temporal questions (date reasoning), and negation questions (absence detection). The critical rule: ground truth must be created by humans who know the documents, NOT by the LLM itself (that would be circular). We start with 50+ questions for statistical validity; with only 3 questions (our current set), a 1-question change causes a 33% swing in metrics — that's not reliable.

"What are the RAGAS metrics and what does each one mean?"
Faithfulness: every claim in the answer must be supported by retrieved context (catches hallucination). Answer Relevancy: the answer must address the actual question (catches off-topic responses). Context Precision: retrieved chunks must be relevant (catches noisy retrieval). Context Recall: all relevant chunks must be retrieved (catches retrieval gaps). You need all four — a system that says "I don't know" scores 1.0 faithfulness but 0.0 relevancy. RAGAS runs these automatically using LLM calls internally.

"How do you monitor cost in a production LLM system?"
Three levers: (1) Cost per request tracked in TokenUsageInfo via Azure OpenAI usage headers. (2) Complexity routing — 85% of queries go to GPT-4o-mini (15x cheaper than GPT-4o), ~69% cost reduction with no quality loss. (3) Semantic cache — similar questions hit the cache (99.9% cost reduction per hit). Budget alerts trigger on daily/hourly thresholds. The key metric to watch is "complex query rate" — if it rises above 40%, the router may be miscalibrated and you're overpaying.

"How do you handle PII in logging and observability?"
Defense in depth: questions are truncated to 100 chars in the controller log. The orchestrator logs entity counts not values ("3 PII items redacted: 2 emails, 1 SSN"). Memory writes and cache writes use PII-redacted text. The API response includes only aggregate counts in PiiSummary. This satisfies GDPR Article 25 (privacy by design) — PII cannot accumulate in Azure Monitor, Log Analytics, or Redis.

"What are reasoning steps and why do you include them in the API response?"
Reasoning steps are an in-band distributed trace — every major pipeline decision is appended to a list and returned with the answer. The client can display: "Intent: DataRetrieval → Called GetSchemaAsync → Called QuerySqlAsync → Routed to GPT-4o (complex) → Reflection score: 8/10". This gives full transparency without a separate APM tool. It also enables client-side debugging — if a user gets a bad answer, they can see exactly which steps the agent took. In production, we additionally ship these steps to Application Insights as structured log events.

"How do you detect quality regression in CI/CD?"
The eval script (run_eval.py) runs as a CI/CD step before every deployment. It calls the live API with a golden test set, runs RAGAS metrics, and fails with exit code 1 if any metric drops below threshold (faithfulness < 0.80, precision/recall/relevancy < 0.70). This means: a bad prompt change, a broken retrieval pipeline, or degraded Azure AI Search index will be caught before reaching production users. The build is blocked. Separately, production quality is monitored via Azure Monitor alerts on the reflection_score metric — a 7-day rolling average below 6.5 triggers a Slack alert.
