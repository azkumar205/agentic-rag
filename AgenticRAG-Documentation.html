2. RAG vs Agentic RAG — Simple Comparison (Azure + .NET)

Understanding the difference is the most common interview topic. Here it is in plain English, then as a table.

🗂️ Classic RAG (Static Pipeline)

How it works:

  1. User sends a question
  2. Question is embedded into a vector
  3. Top-K chunks retrieved from Azure AI Search
  4. Chunks stuffed into an LLM prompt
  5. LLM generates one answer & stops

Limitations:

  • Only one data source (vector store)
  • No planning — just keyword/vector matching
  • No self-evaluation — doesn't know if answer is wrong
  • No memory — forgets previous conversations
  • No caching — every similar question costs LLM tokens
  • Fixed pipeline — cannot adapt to complex questions

🤖 Agentic RAG (Dynamic Pipeline)

How it works:

  1. Check semantic cache — return instantly if similar question seen before
  2. Plan: Planner Agent decides which tools to use and in what order
  3. Execute: Executor Agent calls AI Search, SQL, Document Intelligence, or web search
  4. Reflect: Reflection Agent scores completeness (0.0–1.0)
  5. If score < 0.8 → identify gaps → re-plan (max 2 times)
  6. Cache the final answer in Redis for future similar questions

Advantages:

  • Multiple data sources — search, SQL, PDFs, web
  • Intelligent planning per question type
  • Self-evaluates and retries when uncertain
  • Long-term memory across sessions
  • Semantic cache saves 70-90% of LLM calls for repeat questions
  • Adapts to simple and complex questions equally

One-sentence answer for interviews: Classic RAG is a fixed pipeline (embed → search → generate). Agentic RAG adds Planning (which tools to use), Reasoning (self-evaluation + retry), Memory (persist across sessions), and Caching (reuse previous answers) — making it capable of handling complex multi-step questions from multiple data sources.

Create the file .github/workflows/deploy.yml in the repository with the following content:

YAML — .github/workflows/deploy.yml
# Agentic RAG — CI/CD Pipeline
# Triggers on every push to main:
#   1. Build .NET app and Docker image
#   2. Push image to Azure Container Registry
#   3. Deploy new image to Azure Container Apps

name: Deploy Agentic RAG to Azure

on:
  push:
    branches: [ main ]
  workflow_dispatch:   # allow manual trigger from GitHub UI

env:
  IMAGE_NAME: agentic-rag
  CONTAINER_APP_NAME: agentic-rag-api
  CONTAINER_APP_ENV: cae-agentic-rag

permissions:
  contents: read

jobs:
  build-and-deploy:
    name: Build, Push & Deploy
    runs-on: ubuntu-latest

    steps:
      # ── 1. Checkout source code ─────────────────────
      - name: Checkout repository
        uses: actions/checkout@v4

      # ── 2. Sign in to Azure ─────────────────────────
      - name: Azure Login
        uses: azure/login@v2
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}

      # ── 3. Build & push Docker image to ACR ────────
      - name: Build and push image to ACR
        run: |
          az acr build \
            --registry ${{ secrets.ACR_LOGIN_SERVER }} \
            --image ${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --image ${{ env.IMAGE_NAME }}:latest \
            --file src/AgenticRag/Dockerfile \
            src/AgenticRag

      # ── 4. Create Container Apps environment (idempotent) ──
      - name: Ensure Container Apps environment exists
        run: |
          az containerapp env show \
            --name ${{ env.CONTAINER_APP_ENV }} \
            --resource-group ${{ secrets.AZURE_RG }} \
          || az containerapp env create \
            --name ${{ env.CONTAINER_APP_ENV }} \
            --resource-group ${{ secrets.AZURE_RG }} \
            --location eastus

      # ── 5. Deploy / update Container App ───────────
      - name: Deploy to Azure Container Apps
        run: |
          # Check if the container app already exists
          if az containerapp show \
               --name ${{ env.CONTAINER_APP_NAME }} \
               --resource-group ${{ secrets.AZURE_RG }} > /dev/null 2>&1; then

            # Update existing app with new image
            az containerapp update \
              --name ${{ env.CONTAINER_APP_NAME }} \
              --resource-group ${{ secrets.AZURE_RG }} \
              --image ${{ secrets.ACR_LOGIN_SERVER }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          else
            # Create app on first run
            az containerapp create \
              --name ${{ env.CONTAINER_APP_NAME }} \
              --resource-group ${{ secrets.AZURE_RG }} \
              --environment ${{ env.CONTAINER_APP_ENV }} \
              --image ${{ secrets.ACR_LOGIN_SERVER }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
              --registry-server ${{ secrets.ACR_LOGIN_SERVER }} \
              --target-port 8088 \
              --ingress external \
              --min-replicas 1 \
              --max-replicas 5 \
              --env-vars \
                AzureOpenAI__Endpoint="${{ secrets.AZURE_OPENAI_ENDPOINT }}" \
                AzureOpenAI__ChatDeployment=gpt-4o \
                AzureOpenAI__EmbeddingDeployment=text-embedding-3-large \
                AzureAISearch__Endpoint="${{ secrets.AZURE_SEARCH_ENDPOINT }}" \
                AzureAISearch__IndexName=rag-index \
                DocumentIntelligence__Endpoint="${{ secrets.AZURE_DOCINTEL_ENDPOINT }}"
          fi

      # ── 6. Print the production URL ──────────────────
      - name: Get production URL
        run: |
          APP_URL=$(az containerapp show \
            --name ${{ env.CONTAINER_APP_NAME }} \
            --resource-group ${{ secrets.AZURE_RG }} \
            --query 'properties.configuration.ingress.fqdn' -o tsv)
          echo "✅ Production API live at: https://$APP_URL"
          echo "APP_URL=https://$APP_URL" >> $GITHUB_ENV

5. The Four Agentic Pillars

Every Agentic AI system is built on four capabilities. Understanding these is critical for interviews.

1

🧠 Memory

Short-term: Conversation thread (current session messages). Long-term: Agent Memory store with vector embeddings — persists user preferences and past Q&A across sessions. Implemented in: SemanticCache.cs + Azure AI Foundry memory store.

2

📋 Planning

PlannerAgent uses GPT-4o to decompose a user question into an ordered list of steps, each specifying which tool to call. Example: Step 1 = search_knowledge_base, Step 2 = query_sql, Step 3 = synthesize. Implemented in: Agents/PlannerAgent.cs.

3

🔍 Reasoning

ReflectionAgent evaluates gathered evidence, assigns a confidence score (0.0–1.0), identifies gaps, and decides whether to return the answer or trigger a re-plan. Max 2 re-attempts. Implemented in: Agents/ReflectionAgent.cs.

4

⚡ Action (Tools)

ExecutorAgent dispatches to tools: Azure AI Search, SQL Server, Document Intelligence, and Web Search. Each tool is a focused C# class. Custom tools wrap Azure SDK calls with security validation. Implemented in: Agents/ExecutorAgent.cs + Tools/.

💡 Caching as the 5th Pillar: While not in the classic 4-pillar model, semantic caching is essential in production. SemanticCache.cs uses Redis Enterprise with HNSW vectors so that semantically similar questions (cosine similarity ≥ 0.92) return the cached answer instantly — saving 70–90% of LLM token costs for repeat queries.

6b. From Classic RAG to Agentic RAG — Step by Step

The best way to understand Agentic RAG is to see how it grows from Classic RAG by adding one capability at a time. Each phase below adds a new layer. The code for all phases already exists in this project — phases are conceptual groupings, not separate branches.

Phase 1 — Classic RAG (Baseline)

What you build: The ingestion pipeline + a simple search + LLM answer.

Flow: Upload PDF → IngestionService.IngestFileAsync() → Azure AI Search index is populated. Then query: SearchClient.SearchAsync() → top-K chunks → GPT-4o generates answer.

Key files: Services/IngestionService.cs, Tools/DocumentIntelligenceTool.cs

Limitation: Single search call — if the question needs multiple steps (e.g., compare Q3 vs Q4), classic RAG returns one search result and calls it done. No reasoning about what to look for.

Phase 2 — Add Planning (PlannerAgent)

What changes: Instead of a single search, GPT-4o first decomposes the question into a list of steps (ExecutionPlan) — each step specifies a tool and what to search for.

Flow: Question → PlannerAgent.CreatePlanAsync() → plan with steps[] → executor runs each step.

Key files: Agents/PlannerAgent.cs, Models/ChatModels.cs (ExecutionPlan, PlanStep)

Benefit: Multi-hop queries work. "Compare Q3 and Q4 revenue" becomes two search steps, not one.

Phase 3 — Add Multi-Tool Execution (ExecutorAgent)

What changes: The Executor can dispatch to four different tools based on the plan step: search_knowledge_base, query_sql, extract_document, web_search.

Flow: Each plan step → tool dispatch → evidence gathered → GPT-4o synthesizes.

Key files: Agents/ExecutorAgent.cs, Tools/DocumentIntelligenceTool.cs, Tools/SqlServerTool.cs

Benefit: The system can join knowledge from PDFs (AI Search), databases (SQL), raw files (Document Intelligence), and live web search in a single response.

Phase 4 — Add Reflection & Re-Planning (ReflectionAgent)

What changes: After execution, a ReflectionAgent evaluates the evidence with a confidence score (0–1). If confidence < 0.8, it identifies gaps and triggers a re-plan (up to 2 retries).

Flow: Evidence + question → ReflectionAgent.EvaluateAsync() → {confidence, gaps, needsMoreResearch} → re-plan if needed.

Key files: Agents/ReflectionAgent.cs, Workflow/RagWorkflow.cs

Benefit: The system knows when it doesn't know and tries to fill gaps, rather than hallucinating a confident answer.

Phase 5 — Add Semantic Caching & Memory (SemanticCache)

What changes: A Redis Enterprise HNSW cache stores (question embedding, answer) pairs. Semantically similar questions (cosine ≥ 0.92) return the cached answer instantly without running the full pipeline.

Flow: Question → embed → cosine search in Redis → cache hit → return immediately.

Key files: Cache/SemanticCache.cs, Workflow/RagWorkflow.cs

Benefit: Cost and latency drop dramatically for repeat or similar questions. The system also stores per-user conversation context (long-term memory).

11.1 Tools/DocumentIntelligenceTool.cs — PDF & Image Extraction

Uses the prebuilt-layout model which extracts text from every page AND structured table data. Works on PDFs, JPEGs, PNGs, TIFFs, and more. 

C# — Tools/DocumentIntelligenceTool.cs
using Azure.AI.DocumentIntelligence;
using Azure.Identity;
using System.Text;

namespace AgenticRag.Tools;

public sealed class DocumentIntelligenceTool
{
    private readonly DocumentIntelligenceClient _client;

    public DocumentIntelligenceTool(string endpoint, DefaultAzureCredential credential)
    {
        _client = new DocumentIntelligenceClient(new Uri(endpoint), credential);
    }

    /// <summary>Extract text and tables from a PDF or image file.</summary>
    public async Task<string> ExtractAsync(string filePath)
    {
        await using var stream = File.OpenRead(filePath);

        var operation = await _client.AnalyzeDocumentAsync(
            WaitUntil.Completed,
            "prebuilt-layout",               // prebuilt model — no training needed
            BinaryData.FromStream(stream),
            contentType: "application/octet-stream");

        var result = operation.Value;
        var sb = new StringBuilder();

        // Extract page text line-by-line
        if (result.Pages is not null)
        {
            foreach (var page in result.Pages)
            {
                sb.AppendLine($"--- Page {page.PageNumber} ---");
                if (page.Lines is not null)
                    foreach (var line in page.Lines)
                        sb.AppendLine(line.Content);
            }
        }

        // Extract tables as pipe-delimited rows
        if (result.Tables is not null)
        {
            for (int t = 0; t < result.Tables.Count; t++)
            {
                var table = result.Tables[t];
                sb.AppendLine($"\n--- Table {t + 1} ---");

                var rows = new SortedDictionary<int, SortedDictionary<int, string>>();
                foreach (var cell in table.Cells)
                {
                    if (!rows.ContainsKey(cell.RowIndex))
                        rows[cell.RowIndex] = new SortedDictionary<int, string>();
                    rows[cell.RowIndex][cell.ColumnIndex] = cell.Content;
                }

                foreach (var (_, cols) in rows)
                {
                    var values = Enumerable.Range(0, table.ColumnCount)
                        .Select(c => cols.TryGetValue(c, out var v) ? v : "")
                        .ToArray();
                    sb.AppendLine(string.Join(" | ", values));
                }
            }
        }

        return sb.ToString();
    }
}

11.2 Tools/SqlServerTool.cs — Read-Only SQL Queries

🔒 Security: Only SELECT queries are permitted. A keyword blocklist prevents INSERT, UPDATE, DELETE, DROP, ALTER, EXEC, and all other write/DDL operations. Results are limited to 100 rows to prevent data exfiltration. 
C# — Tools/SqlServerTool.cs
using Microsoft.Data.SqlClient;
using System.Text;

namespace AgenticRag.Tools;

public sealed class SqlServerTool
{
    private readonly string _connectionString;

    // Keywords that indicate write or dangerous operations — BLOCK all of them
    private static readonly HashSet<string> BlockedKeywords = new(StringComparer.OrdinalIgnoreCase)
    {
        "INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "CREATE",
        "TRUNCATE", "EXEC", "EXECUTE", "MERGE", "GRANT", "REVOKE",
        "xp_", "sp_"
    };

    private const int MaxRows = 100;

    public SqlServerTool(string connectionString) => _connectionString = connectionString;

    /// <summary>Execute a read-only SELECT query. Returns pipe-delimited table.</summary>
    public async Task<string> QueryAsync(string query)
    {
        var trimmed = query.Trim();

        // Rule 1: Must start with SELECT
        var firstWord = trimmed.Split(' ', StringSplitOptions.RemoveEmptyEntries)
                              .FirstOrDefault()?.ToUpper();
        if (firstWord != "SELECT")
            return "Error: Only SELECT queries are allowed.";

        // Rule 2: Block dangerous keywords
        var upper = query.ToUpper();
        foreach (var kw in BlockedKeywords)
            if (upper.Contains(kw, StringComparison.OrdinalIgnoreCase))
                return $"Error: '{kw}' operations are not permitted.";

        await using var connection = new SqlConnection(_connectionString);
        await connection.OpenAsync();
        await using var command = new SqlCommand(trimmed, connection) { CommandTimeout = 30 };
        await using var reader = await command.ExecuteReaderAsync();

        var sb = new StringBuilder();

        // Column headers
        var columns = Enumerable.Range(0, reader.FieldCount).Select(reader.GetName).ToArray();
        sb.AppendLine(string.Join(" | ", columns));
        sb.AppendLine(new string('-', sb.Length));

        // Rows — max 100
        int rowCount = 0;
        while (await reader.ReadAsync() && rowCount < MaxRows)
        {
            var values = Enumerable.Range(0, reader.FieldCount)
                .Select(i => reader.IsDBNull(i) ? "NULL" : reader.GetValue(i).ToString() ?? "")
                .ToArray();
            sb.AppendLine(string.Join(" | ", values));
            rowCount++;
        }

        return sb.ToString();
    }
}

13. Planning Pillar — Agents/PlannerAgent.cs

The Planner Agent receives the user's question and uses GPT-4o to produce a structured JSON plan — an ordered list of steps specifying which tool to call at each stage. This is what makes the system "agentic": it adapts the retrieval strategy to the question.

How Planning Works — Step by Step

  1. User question arrives (e.g., "Compare Q3 and Q4 revenue and check for discrepancies")
  2. PlannerAgent sends the question + system prompt to GPT-4o with ResponseFormat = JSON
  3. GPT-4o returns a JSON plan: which tools to use and in what order
  4. The plan is deserialized into ExecutionPlan (list of PlanStep records)
  5. The plan is passed to the ExecutorAgent for execution
C# — Agents/PlannerAgent.cs
using Azure.AI.OpenAI;
using Azure.Identity;
using OpenAI.Chat;
using AgenticRag.Models;
using System.Text.Json;

namespace AgenticRag.Agents;

public sealed class PlannerAgent
{
    private readonly ChatClient _chatClient;

    // The system prompt tells GPT-4o what tools are available and how to respond
    private const string SystemPrompt = @"You are a planning agent. Given a user question,
decompose it into ordered steps.

Available tools:
- ""search_knowledge_base"": Search internal documents via Azure AI Search
- ""query_sql"": Query SQL Server (generate SELECT query only)
- ""extract_document"": Extract text from a specific PDF or image file path
- ""web_search"": Search the internet for real-time or external information

Rules:
1. Prefer internal sources before web_search
2. For simple questions, 1-2 steps is enough
3. Always end with a step using toolToUse: ""synthesize""

Respond ONLY with JSON:
{ ""steps"": [{ ""stepNumber"": 1, ""description"": ""..."", ""toolToUse"": ""..."", ""toolInput"": ""..."" }] }";

    public PlannerAgent(string endpoint, string deployment, DefaultAzureCredential credential)
    {
        var client = new AzureOpenAIClient(new Uri(endpoint), credential);
        _chatClient = client.GetChatClient(deployment);
    }

    /// <summary>Create an execution plan. Pass gapInfo on re-plan attempts.</summary>
    public async Task<ExecutionPlan> CreatePlanAsync(string question, string? gapInfo = null)
    {
        var messages = new List<ChatMessage>
        {
            new SystemChatMessage(SystemPrompt),
            new UserChatMessage(
                gapInfo is null
                    ? $"Question: {question}"
                    : $"Question: {question}\nGap from previous attempt: {gapInfo}")
        };

        var response = await _chatClient.CompleteChatAsync(messages, new ChatCompletionOptions
        {
            ResponseFormat = ChatResponseFormat.CreateJsonObjectFormat(), // Force JSON output
            Temperature = 0.2f  // Low temperature = deterministic planning
        });

        var json = response.Value.Content[0].Text;
        var plan = JsonSerializer.Deserialize<ExecutionPlan>(json, new JsonSerializerOptions
        {
            PropertyNameCaseInsensitive = true
        });

        return plan ?? new ExecutionPlan(new List<PlanStep>());
    }
}

14. Reasoning Pillar — Agents/ReflectionAgent.cs

After the Executor Agent gathers evidence, the Reflection Agent evaluates it. This is the "reasoning" step — the system checks its own work before returning an answer. If the confidence score is below 0.8, it triggers a re-plan with specific gap information.

          │

  ReflectionAgent evaluates:

15. Caching Pillar — Cache/SemanticCache.cs

Semantic caching is different from regular key-value caching. Instead of caching exact strings, we cache by meaning. Two questions with the same meaning but different wording will get the same cached answer.

How Semantic Caching Works

ConceptExplanation
Vector EmbeddingThe question is converted to a 3072-float vector using text-embedding-3-large. Similar questions produce similar vectors.
HNSW IndexRedis Enterprise stores all question vectors in an HNSW (Hierarchical Navigable Small World) index for fast approximate nearest-neighbor search.
Cosine SimilarityMeasures how "close" two vectors are. Value 0.0–1.0, where 1.0 = identical. We use threshold 0.92 (configurable).
Cache HitIf the nearest cached question has similarity ≥ 0.92, return the cached answer. Zero LLM calls, sub-100ms response.
Cache MissRun the full agentic pipeline, then store the new Q&A pair for future queries.
 
C# — Cache/SemanticCache.cs (key methods)
using Azure.AI.OpenAI;
using Azure.Identity;
using OpenAI.Embeddings;
using StackExchange.Redis;
using System.Text.Json;

namespace AgenticRag.Cache;

public sealed class SemanticCache
{
    private readonly IDatabase _db;
    private readonly EmbeddingClient _embeddingClient;
    private readonly double _threshold;
    private const string IndexName = "semantic_cache_idx";
    private const string Prefix = "cache:";

    public SemanticCache(string redisConn, string oaiEndpoint,
        string embDeployment, int dims, double threshold, DefaultAzureCredential credential)
    {
        var redis = ConnectionMultiplexer.Connect(redisConn);
        _db = redis.GetDatabase();
        _threshold = threshold;
        var oai = new AzureOpenAIClient(new Uri(oaiEndpoint), credential);
        _embeddingClient = oai.GetEmbeddingClient(embDeployment);
        EnsureIndex(dims);
    }

    /// <summary>Look up a semantically similar question. Returns null on miss.</summary>
    public async Task<string?> LookupAsync(string question)
    {
        var embedding = await EmbedAsync(question);
        var vecBytes = FloatsToBytes(embedding);

        // KNN = K nearest neighbors — find the 1 most similar cached question
        var result = await _db.ExecuteAsync("FT.SEARCH", IndexName,
            "(*)=>[KNN 1 @embedding $vec AS score]",
            "PARAMS", "2", "vec", vecBytes,
            "RETURN", "3", "question", "answer", "score",
            "SORTBY", "score", "DIALECT", "2");

        if (result is null) return null;
        var arr = (RedisResult[])result!;
        if ((long)arr[0] == 0) return null; // No results at all

        var fields = (RedisResult[])arr[2];
        string? answer = null;
        double score = 1.0;

        for (int i = 0; i < fields.Length; i += 2)
        {
            if ((string)fields[i]! == "score")  score  = double.Parse((string)fields[i+1]!);
            if ((string)fields[i]! == "answer") answer = (string)fields[i+1]!;
        }

        // COSINE distance (0=identical, 1=opposite) → convert to similarity
        var similarity = 1.0 - score;
        return similarity >= _threshold ? answer : null;
    }

    /// <summary>Store a question + answer with its embedding for future lookups.</summary>
    public async Task StoreAsync(string question, string answer)
    {
        var embedding = await EmbedAsync(question);
        var key = $"{Prefix}{Guid.NewGuid():N}";
        var json = JsonSerializer.Serialize(new { question, answer, embedding });
        await _db.ExecuteAsync("JSON.SET", key, "$", json);
    }

    private async Task<float[]> EmbedAsync(string text)
    {
        var response = await _embeddingClient.GenerateEmbeddingAsync(text);
        return response.Value.ToFloats().ToArray();
    }

    private static byte[] FloatsToBytes(float[] floats)
    {
        var bytes = new byte[floats.Length * sizeof(float)];
        Buffer.BlockCopy(floats, 0, bytes, 0, bytes.Length);
        return bytes;
    }

    private void EnsureIndex(int dims)
    {
        try { _db.Execute("FT.INFO", IndexName); }
        catch
        {
            // Create HNSW vector index if it doesn't exist
            _db.Execute("FT.CREATE", IndexName, "ON", "JSON",
                "PREFIX", "1", Prefix, "SCHEMA",
                "$.question", "AS", "question", "TEXT",
                "$.answer", "AS", "answer", "TEXT",
                "$.embedding", "AS", "embedding", "VECTOR", "HNSW", "6",
                "TYPE", "FLOAT32", "DIM", dims.ToString(), "DISTANCE_METRIC", "COSINE");
        }
    }
}

16. Workflow Orchestrator — Workflow/RagWorkflow.cs

The RagWorkflow is the central coordinator. It wires all three agents together: Planner → Executor → Reflection, with a conditional re-plan loop (max 2 attempts). This is registered as a singleton in Program.cs DI container.

The Complete Pipeline in Code

C# — Workflow/RagWorkflow.cs
using AgenticRag.Agents;
using AgenticRag.Cache;
using AgenticRag.Models;
using Microsoft.Extensions.Logging;

namespace AgenticRag.Workflow;

public sealed class RagWorkflow
{
    private readonly SemanticCache _cache;
    private readonly PlannerAgent _planner;
    private readonly ExecutorAgent _executor;
    private readonly ReflectionAgent _reflector;
    private readonly ILogger<RagWorkflow> _logger;

    private const int MaxReplanAttempts = 2; // Max re-planning iterations

    public RagWorkflow(
        SemanticCache cache, PlannerAgent planner,
        ExecutorAgent executor, ReflectionAgent reflector,
        ILogger<RagWorkflow> logger)
    {
        _cache = cache; _planner = planner;
        _executor = executor; _reflector = reflector; _logger = logger;
    }

    /// <summary>
    /// Full agentic pipeline:
    ///   1. Check semantic cache → return instantly if hit
    ///   2. Plan → which tools to call in what order
    ///   3. Execute → gather evidence from all tools
    ///   4. Reflect → confidence score + gap analysis
    ///   5. Re-plan if confidence < 0.8 (max 2 times)
    ///   6. Cache final answer + return
    /// </summary>
    public async Task<ChatResponse> AskAsync(string question, string userId)
    {
        // ── Step 1: Semantic Cache ──────────────────────────────────────
        var cached = await _cache.LookupAsync(question);
        if (cached is not null)
        {
            _logger.LogInformation("Cache HIT — returning instantly");
            return new ChatResponse(cached, FromCache: true,
                PlanSteps: 0, ReflectionAttempts: 0, ConfidenceScore: 1.0);
        }

        // ── Step 2: Plan ───────────────────────────────────────────────
        var plan = await _planner.CreatePlanAsync(question);
        _logger.LogInformation("Plan: {StepCount} steps", plan.Steps.Count);

        // ── Steps 3–5: Execute + Reflect loop ─────────────────────────
        string? finalAnswer = null;
        double confidence = 0;
        int reflectionAttempts = 0;

        for (int attempt = 0; attempt <= MaxReplanAttempts; attempt++)
        {
            reflectionAttempts = attempt + 1;

            // Execute: run all plan steps, gather evidence
            var evidence = await _executor.ExecuteAsync(question, plan);
            _logger.LogInformation("Execution done (attempt {A})", attempt + 1);

            // Reflect: evaluate evidence quality
            var reflection = await _reflector.EvaluateAsync(question, evidence);
            confidence = reflection.ConfidenceScore;
            _logger.LogInformation("Confidence: {Score:F2}, Complete: {Done}",
                reflection.ConfidenceScore, reflection.IsComplete);

            if (reflection.IsComplete && reflection.FinalAnswer is not null)
            {
                finalAnswer = reflection.FinalAnswer;
                break; // 🎉 Good enough answer found!
            }

            // Re-plan with gap information (if we have attempts remaining)
            if (attempt < MaxReplanAttempts && reflection.SuggestedAction is not null)
            {
                _logger.LogInformation("Re-planning — gap: {Gap}", reflection.GapAnalysis);
                var gapInfo = $"{reflection.GapAnalysis}. Try: {reflection.SuggestedAction}";
                plan = await _planner.CreatePlanAsync(question, gapInfo);
            }
        }

        finalAnswer ??= "Unable to find a satisfactory answer after multiple attempts.";

        // ── Step 6: Cache the result ───────────────────────────────────
        await _cache.StoreAsync(question, finalAnswer);
        _logger.LogInformation("Answer cached for future similar questions");

        return new ChatResponse(finalAnswer, FromCache: false,
            PlanSteps: plan.Steps.Count,
            ReflectionAttempts: reflectionAttempts,
            ConfidenceScore: confidence);
    }
}

18. Data Ingestion — Loading Documents into Azure AI Search

Before the agent can search your internal documents, you need to ingest them into Azure AI Search. The full ingestion pipeline is implemented in src/AgenticRag/Services/IngestionService.cs: Extract text (Document Intelligence) → Chunk → Embed (text-embedding-3-large) → Upload to AI Search index.

The POST /api/ingest endpoint in Program.cs exposes this pipeline over HTTP. Upload any PDF, DOCX, or image — the endpoint creates the index if needed, chunks and embeds the document, and uploads everything to Azure AI Search. It returns the number of chunks created.

Ingestion Steps (all implemented in IngestionService.cs)

EnsureIndexExistsAsync() — Create the Azure AI Search Index

Defines fields: id (key), content (searchable), source (filterable), title (text), content_vector (3072-dim vector with HNSW profile + semantic config). Uses SearchIndexClient.CreateOrUpdateIndexAsync() — safe to call on every startup (idempotent).

IngestFileAsync() — Step 1: Extract Text

Calls DocumentIntelligenceTool.ExtractAsync(filePath) — works for PDF, DOCX, JPEG, PNG, TIFF. Returns plain text including table content extracted from every page.

IngestFileAsync() — Step 2: Chunk the Text

Calls ChunkText(text, size=1000, overlap=200). Overlap ensures sentences at chunk boundaries are not lost when searching. Each chunk is independently searchable.

IngestFileAsync() — Step 3: Embed Each Chunk

Calls EmbeddingClient.GenerateEmbeddingAsync(chunk) → 3072-float vector (text-embedding-3-large). Each chunk gets its own vector for semantic similarity search.

IngestFileAsync() — Step 4: Upload to Azure AI Search

Calls SearchClient.UploadDocumentsAsync(batch) in batches of 100. Azure AI Search indexes all fields automatically (BM25 for keyword, HNSW for vector, semantic for hybrid).

C# — src/AgenticRag/Services/IngestionService.cs (full implementation)
using Azure.AI.OpenAI;
using Azure.Identity;
using Azure.Search.Documents;
using Azure.Search.Documents.Indexes;
using Azure.Search.Documents.Indexes.Models;
using Azure.Search.Documents.Models;
using OpenAI.Embeddings;
using AgenticRag.Tools;

namespace AgenticRag.Services;

public sealed class IngestionService
{
    private readonly DocumentIntelligenceTool _docTool;
    private readonly EmbeddingClient _embeddingClient;
    private readonly SearchClient _searchClient;
    private readonly SearchIndexClient _indexClient;
    private readonly string _indexName;
    private readonly int _embeddingDimensions;

    // ... constructor omitted for brevity — see full file in src/AgenticRag/Services/IngestionService.cs

    /// Ensures AI Search index exists with correct schema (idempotent)
    public async Task EnsureIndexExistsAsync()
    {
        var fields = new List<SearchField>
        {
            new SimpleField("id", SearchFieldDataType.String) { IsKey = true },
            new SearchableField("content"),
            new SimpleField("source", SearchFieldDataType.String) { IsFilterable = true },
            new SearchableField("title"),
            new VectorSearchField("content_vector", _embeddingDimensions, "hnsw-config")
        };
        // ... HNSW algorithm + semantic config setup (see IngestionService.cs)
        await _indexClient.CreateOrUpdateIndexAsync(new SearchIndex(_indexName) { Fields = fields, /* ... */ });
    }

    /// Full ingestion: Extract → Chunk → Embed → Upload (returns chunk count + chars)
    public async Task<IngestionResult> IngestFileAsync(string filePath, int chunkSize = 1000, int overlap = 200)
    {
        // Step 1: Extract text via Document Intelligence
        var text = await _docTool.ExtractAsync(filePath);

        // Step 2: Chunk with overlap
        var chunks = ChunkText(text, chunkSize, overlap).ToList();

        // Step 3: Embed + build SearchDocument list
        var docs = new List<SearchDocument>(chunks.Count);
        for (int i = 0; i < chunks.Count; i++)
        {
            var embedding = await _embeddingClient.GenerateEmbeddingAsync(chunks[i]);
            docs.Add(new SearchDocument
            {
                ["id"] = Guid.NewGuid().ToString("N"),
                ["content"] = chunks[i],
                ["source"] = filePath,
                ["title"] = $"{Path.GetFileName(filePath)} - chunk {i + 1}",
                ["content_vector"] = embedding.Value.ToFloats().ToArray()
            });
        }

        // Step 4: Upload in batches of 100
        for (int i = 0; i < docs.Count; i += 100)
            await _searchClient.UploadDocumentsAsync(docs.GetRange(i, Math.Min(100, docs.Count - i)));

        return new IngestionResult(Path.GetFileName(filePath), docs.Count, text.Length);
    }

    private static IEnumerable<string> ChunkText(string text, int size, int overlap)
    {
        for (int start = 0; start < text.Length; start += size - overlap)
            yield return text[start..Math.Min(start + size, text.Length)];
    }
}

22. Interview Ready Points — .NET Agentic RAG on Azure

These are the most common interview questions about Agentic AI systems with Azure + .NET. Read the question, understand the answer, then practice saying it in your own words.

🎯 Core Concepts

Q: What is the difference between RAG and Agentic RAG?

Classic RAG is a fixed pipeline: embed question → vector search → LLM generates answer. Agentic RAG adds four capabilities: Planning (choosing which tools to use), Reasoning (evaluating if the answer is good enough and retrying), Memory (remembering past conversations), and Caching (reusing answers for similar questions). The key difference is adaptability — Agentic RAG can query SQL, search PDFs, use the web, AND reflect on whether its answer is complete.

Q: What are the four pillars of Agentic AI?

Memory (short-term: conversation thread; long-term: semantic cache + user profile), Planning (decomposing questions into tool-specific steps), Reasoning (self-evaluation with confidence scoring and gap analysis), and Action (tool execution: AI Search, SQL, Document Intelligence, Web Search).

Q: How does semantic caching work and why is it important?

Semantic caching stores Q&A pairs with vector embeddings in Redis. When a new question arrives, it's embedded and compared to cached questions using cosine similarity. If similarity ≥ 0.92, the cached answer is returned instantly — no LLM API call needed. This is important because: (1) it drastically reduces Azure OpenAI costs for repeat/similar questions, (2) sub-100ms response vs. 3–10 seconds for LLM calls, and (3) it serves as the system's long-term memory.

Q: How does the reflection/reasoning loop prevent bad answers?

After the Executor Agent gathers evidence, the Reflection Agent evaluates it and assigns a confidence score (0.0–1.0). If confidence < 0.8, it produces a gap analysis ("Q4 data missing") and a suggested action ("try web_search"). The Planner Agent re-plans using this gap info. This loop runs at most 2 times (configurable). If confidence ≥ 0.8, the answer is returned. The key insight: the agent knows when it doesn't know enough.

🔧 Technical Implementation (.NET)

Q: How is DefaultAzureCredential used in production vs. development?

In development, DefaultAzureCredential picks up credentials from az login (Azure CLI token). In production (Container Apps/AKS), it uses the pod's Managed Identity — no API keys or passwords needed in the code. You grant the managed identity RBAC roles (e.g., "Cognitive Services OpenAI User") and Azure handles authentication automatically. This is the recommended zero-secret approach.

Q: Why use ASP.NET Core Minimal API instead of controllers?

Minimal API requires less boilerplate — no controllers, no attribute routing setup, no separate controller classes. For an agent API with just 2–3 endpoints, it's cleaner. The app.MapPost("/api/chat", async (ChatRequest request, RagWorkflow workflow) => ...) pattern is readable and the DI injection works the same way as with controllers.

Q: How is SQL injection prevented in SqlServerTool?

Two layers: (1) The query must start with SELECT — any other first word is rejected. (2) A blocklist of dangerous keywords (INSERT, UPDATE, DELETE, DROP, EXEC, ALTER, etc.) is checked against the entire query. Results are limited to 100 rows. Note: this is defense-in-depth for LLM-generated queries; for user-facing SQL, always use parameterized queries.

Q: Why use C# records for data models instead of classes?

Records are immutable by default (value-based equality, no accidental mutation), which is ideal for DTOs passed between agents. The compiler generates ToString() for free (great for logging), and the concise positional syntax (record PlanStep(int StepNumber, string ToolToUse, ...)) reduces boilerplate significantly.

Q: How does multi-stage Dockerfile reduce production image size?

Stage 1 uses mcr.microsoft.com/dotnet/sdk:8.0 (full SDK, ~800MB) to restore packages and compile. Stage 2 uses mcr.microsoft.com/dotnet/aspnet:8.0 (runtime only, ~200MB) and copies only the published output. The final image is ~200MB instead of ~800MB. This also means compiler tools (and potential attack surface) are not in the production image.

🏗️ Architecture & Design

Q: Why register agents as singletons in DI instead of scoped/transient?

Agents wrap AzureOpenAIClient which manages connection pooling. Creating a new client per request is expensive. Singletons are created once and reused, amortizing the connection setup cost across all requests. The agents hold no per-request state — all state is passed as method parameters — so singletons are safe.

Q: What happens if the LLM returns malformed JSON in the plan or reflection?

Both PlannerAgent and ReflectionAgent use ResponseFormat = ChatResponseFormat.CreateJsonObjectFormat() which instructs the model to always output valid JSON. If deserialization still fails (model returns unexpected structure), the null-coalescing operator returns safe defaults: an empty plan or a failed reflection with IsComplete=false. The workflow handles this gracefully by using the fallback answer.

Q: How would you scale this system to handle 1000 concurrent users?

Key scaling points: (1) Semantic cache — similar questions are served from Redis, dramatically reducing LLM load. (2) Container Apps auto-scaling — --min-replicas 1 --max-replicas 20. (3) Azure OpenAI throttling — use provisioned throughput (PTU) deployments instead of pay-as-you-go. (4) Redis clustering — Enterprise tier already supports clustering. (5) Async throughout — all C# code uses async/await, so threads are not blocked on I/O.

🚀 Azure & DevOps

Q: What Azure services are used and why each one?

Azure OpenAI: GPT-4o for chat/planning/reflection, text-embedding-3-large for vectors. Azure AI Search: hybrid vector + keyword + semantic search for internal documents. Document Intelligence: OCR + table extraction from PDFs/images (no custom ML needed). SQL Server/Azure SQL: structured data queries (orders, revenue, metrics). Redis Enterprise: semantic cache (HNSW vector index requires Enterprise tier). Container Apps: serverless containers with auto-scaling and managed identity.

Q: How would you add CI/CD for this project?

GitHub Actions workflow: (1) on push to main: dotnet build + dotnet test. (2) If tests pass: az acr build to push new Docker image. (3) az containerapp update --image new-tag for zero-downtime rolling deployment. Add branch protection requiring tests to pass before merge. Use Azure Key Vault for any remaining secrets.
